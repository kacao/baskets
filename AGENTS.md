# CLAUDE.md

Baskets — multi-tenant project management app (organizations → workspaces → projects → tasks → sub-tasks; ADR-062). Minimalist. UI built on Tailwind 4 + DaisyUI 5 (ADR: Cross-cutting). Product scope: `PRD.md`. Architecture decisions: `ADR.md`.

## Commands

```bash
npm run dev          # dev server on :5173 (vite.config `server.host: true` binds all interfaces so a LAN/tailnet host can reach it; `server.allowedHosts` lists non-localhost hostnames Vite will serve, e.g. `inspiron`)
npm run build        # production build (adapter-node → build/)
npm run check        # svelte-kit sync + svelte-check (run before committing)
npm run verify       # fast pre-commit gate = check + test:unit (server-free; ADR-058)
npm run lint         # ESLint flat config (permissive — warnings allowed, 0 errors; ADR-059)
npm run format       # prettier --write .   (format:check = verify formatting, no write)
npm run db:push      # apply the active dialect's schema to the DB (drizzle-kit push; dialect+creds from .env — ADR-050)
npm run db:seed      # seed admin/demo users + sample data (idempotent; skips if admin exists)
npm run start        # serve production build via custom server.js (adapter-node handler + /ws WebSocket; reads .env)
```

Tests: **vitest** — `npm run test:unit` (pure modules in `tests/unit/`), `npm test` (all, incl. `*.svelte.test.ts` component tests — note `test:unit` filters to `tests/unit/` so it does NOT pick those up), `npm run test:coverage` (v8 report). **`npm run test:server`** = server service-layer characterization tests in `tests/server/` against an **isolated per-run SQLite DB** (ADR-060) — never touches the dev DB. **`npm run test:integration` + `npm run test:e2e`** hit a running seeded dev server and are gated behind **`RUN_INTEGRATION=1`** (ADR-058) so a missing server FAILS loudly instead of skipping-as-passing. Add unit specs for new pure `$lib` helpers (e.g. `mentions`/`files`/`customFields`/`taskFilter`/`date`/`taskGroups`); add a `tests/server/` case for new service-layer logic. Always run `npm run verify` before committing; still smoke-test UI changes manually.

db:push gotcha: when one push both rebuilds a table (e.g. dropping a unique) AND adds a column to it, drizzle-kit generates a broken `INSERT … SELECT new_column FROM old_table` and dies mid-push, leaving the DB partially migrated. Back up the .db first; if it happens, apply the missing `ALTER TABLE … ADD COLUMN` by hand, then re-run `npx drizzle-kit push --force` until "Changes applied".

## Stack

- SvelteKit 2 + **Svelte 5 runes** (`$props`, `$state`, `$derived` — no legacy `export let` / `$:`)
- UI: Tailwind 4 (`@tailwindcss/vite`) + DaisyUI 5 (see Theming under Conventions)
- **Multi-dialect DB via Drizzle ORM (ADR-050)** — `DB_DIALECT` in `.env` selects `sqlite` (default, `better-sqlite3`) or `postgres` (`postgres-js`); `DATABASE_URL` is the connection (file path for sqlite [default `./data/baskets.db`, in the gitignored `data/` dir kept via `data/.gitkeep`]; `postgres://…` for pg). MySQL is deferred (structure ready). New deploys just set the two env vars + `npm run db:push`.
- BetterAuth 1.6 with `twoFactor` (TOTP), `admin`, and `organization` plugins (ADR-062: org roles live ONLY on `member.role`, never `user.role`; teams/dynamicAccessControl disabled; plugin org-deletion disabled — deletion goes through `deleteOrganizationGuarded`)

## Architecture

- `src/lib/server/db/` — **multi-dialect schema (ADR-050)**. `schema.ts` is a thin FACADE that re-exports all 35 tables from the active dialect (so importers keep `import { task } from '$lib/server/db/schema'`). The real definitions live in **`schema.sqlite.ts` (canonical)** and **`schema.pg.ts`** — byte-identical table/column NAMES, only types differ; **edit BOTH in lockstep** for any schema change. `dialect.ts` = single dialect source of truth (reads `process.env.DB_DIALECT`, script-safe). `index.ts` picks driver+schema and exports `db` (typed off the sqlite schema; the PG instance is cast to it — the query surface is compatible). Tables = BetterAuth (user, session, account, verification, twoFactor, organization, member, invitation) AND app tables (workspace, project, task, …). BetterAuth field requirements change between versions; if it errors about a missing field, add the column to BOTH schema files and `npm run db:push`. Keep `scripts/seed.ts`'s auth config in sync with `src/lib/server/auth.ts` (both map the adapter `provider` off `DIALECT`).
- Organizations (ADR-062) — the TENANT, one level above workspaces. Tables `organization`/`member`/`invitation` (BetterAuth organization plugin; `workspace.organizationId` FK). Org service = `src/lib/server/orgs.ts`: `resolveActiveOrgContext` (the ONE org→workspace resolver for loads), `orgRole`/`isOrgAdmin`, `createWorkspaceService`, `createOrganizationForUser`, `deleteOrganizationGuarded` (owner-only, 0 workspaces, explicit transactional sweep — the ALTER-added org columns have no real FK on sqlite), invite/accept/cancel/remove/role/leave services, `purgeStaleGrants`. Active org = **`org` cookie** (validated against memberships, silent fallback = first membership; the app ignores `session.activeOrganizationId`); switching org clears the `workspace` cookie; **project-page loads auto-align the org cookie** to an accessible project's org (makes bell clicks/deep links coherent). UI: org switcher in the sidebar header (above the workspace switcher); `/onboarding` (0-org users are redirected here by the (app) layout; also `?new=1` for extra orgs); `/orgs/[id]/settings` (General/Members/Invitations/Danger; non-member → 404); **`/invite/[id]`** top-level accept route (works signed-out; the link IS the capability — there is deliberately NO by-email invite discovery while registration is unverified; `?redirect=` params are validated same-origin-relative via `src/lib/safeRedirect.ts`). REST: `/api/orgs` (+`/{id}`, `/members[/{userId}]`, `/invites[/{invId}]`), `POST /api/invites/{id}/accept`; `GET /api/me` gains `orgs[]` (attach.js's parse preserved). `dispatchEvent` resolves the org's integration row via the event's project (`integration` is per-org: `unique(organizationId, type)`, org required in code). Notifications carry `organizationId` (required at create); the bell spans the user's member orgs with an org badge on non-active rows. Boot migration `ensureDefaultOrganization` (in `workspaces.ts`) is one-time + marker-gated (`organization.metadata {"migrated":true}`), crash-resumable, fresh-install no-op; post-marker a null `workspaceId`/`organizationId` is a logged integrity error, never adopted.
- Workspaces: every project belongs to one (`project.workspaceId`), every workspace to one org; workspaces own custom statuses and ALL labels/groups. `workspace.ownerId` + `permission` grants (`resourceType 'workspace'`) gate workspace structure edits and cascade to its projects/views — and per ADR-019/062 they also define VISIBILITY within the org (plain members only see workspaces they own/are granted, plus directly granted projects; org owners/admins see all of their org). Workspace creation goes through `createWorkspaceService` (org membership required, org stamped, per-org name uniqueness); the nullable `workspaceId`/`organizationId` FKs exist only for db:push on populated tables — code treats them as required. Workspace CRUD at /workspaces; per-workspace settings pane (general/statuses/labels/grants/delete) at /workspaces/[id]/settings. A workspace is deletable only when empty — even the last one (a 0-workspace org is legal; org deletion requires it). A user's FIRST workspace (they own none yet — `isFirstWorkspaceForUser`; org bootstrap uses `shouldSeedSamplesForOwner`) is seeded with sample projects/milestones/tasks via `seedWorkspaceSamples` (in `projects.ts`). The sidebar header is the org switcher + workspace switcher (current names + chevrons; `org`/`workspace` cookies, both validated with fallback) and the sidebar project list is scoped to the current workspace. **"Projects" is a section label** (not a link); each project is a **collapsible menu** (chevron, auto-expands the active project) → **Overview / Tasks / Milestones / Statuses / Labels / Custom fields / Locations / Files / Settings** sub-items with icons (Heroicons, ADR-052). Each is its own route under `/projects/[id]/*`: the 5 structure sub-pages (`milestones`/`statuses`/`labels`/`custom-fields`/`locations`) **re-export the settings `load`+`actions`** (`ProjectSettingsData` shared type — single source of truth; their forms post `?/action` to their own route), and `/projects/[id]/settings` is trimmed to General/Depends/Grants/Danger + an **Import / Export** section (ADR-039). **Overview + Files (ADR-046)** are instead **independent pages** gated by `canAccessProject` (→404, NOT the settings `canEditProject`/403 load) so viewers see them read-only: **Overview** (`/projects/[id]/overview`) = a borderless, document-like title+description editor (title font > description, auto-save on blur, reuses the settings `updateProject` action — which now also broadcasts + runs mention notifications); **Files** (`/projects/[id]/files`) = an aggregate of every project file (task attachments + custom-field uploads + project-level uploads) attributed to its source, with search/type/source filters, grid⇄list, image lightbox, download, delete, and a drag-drop upload zone backed by `GET`/`POST /api/projects/[id]/files` (project-level file = `taskId`+`fieldId` both null; mirrors `POST /api/files` validation; `storagePath` never returned).
- `src/lib/server/auth.ts` — BetterAuth instance (incl. `organizationHooks`: server-generated slug, idempotent workspace bootstrap, grant purge + WS kick on member removal/re-join). `src/lib/auth-client.ts` — browser client (used for auth + admin actions). `src/hooks.server.ts` — runs `ensureDefaultStatuses` + the one-time `ensureDefaultOrganization` migration, then populates `locals.user` / `locals.session` on every request.
- Routes: `(auth)` group = login/register/two-factor (redirects away if signed in — honoring a validated same-origin `?redirect=` param, e.g. from `/invite/[id]`); `(app)` group = everything else (redirects to /login if not; 0-org users additionally redirect to /onboarding, which is exempt; its `+layout.svelte` is the shell with sidebar). `/invite/[id]` is top-level (works signed-out). `/admin` additionally requires `locals.user.role === 'admin'` (instance operator; enforced in its `+page.server.ts`).
- Data mutations: SvelteKit form actions in `+page.server.ts` files (projects, tasks) — thin adapters over the shared service layer (see REST API below; ADR-049). Exception: auth/2FA/admin-user actions go through `authClient` (BetterAuth API) client-side.
- REST API (ADR-036 — **full UI→API parity**): `src/routes/api/**/+server.ts`. Auth via session cookie OR `Authorization: Bearer bsk_…` API key. Keys are managed per-user in /settings (SHA-256 hash stored in `api_key` table, plaintext shown once); `src/lib/server/api-keys.ts` generates/resolves them, `src/hooks.server.ts` resolves the bearer header into `locals.user`. Shared validation helpers in `src/lib/server/api.ts`. **Every web-UI form action now has a REST endpoint** (resource-oriented): projects, tasks (+ `bulk`, `move` [board order — `POST /api/tasks/{id}/move` over `moveTaskService`], `labels`, `dependencies`), comments, files, notifications, templates (+ `PATCH`/`DELETE /api/templates/{id}`), export, me — PLUS **orgs** (+ nested `members[/{userId}]`/`invites[/{invId}]`, and `POST /api/invites/{id}/accept`; ADR-062), workspaces (+ nested `statuses`/`labels`/`label-groups`/`grants`; collections take `?org=`, POST takes `organizationId`), `statuses/{id}`, `labels/{id}`, `label-groups/{id}`, project `statuses`/`labels`/`milestones`/`locations`/`views`/`custom-fields`/`custom-values`/`dependencies`/`grants`, `milestones/{id}`(+`/dependencies`), `locations/{id}`, `views/{id}`, `custom-fields/{id}`(+`/options`), `custom-field-options/{id}`, `keys`. EXCLUDED on purpose: **Integrations** (write-only Slack secret, UI-only), auth/account/2FA (BetterAuth), and project **Import** (ADR-039: web-only by design — the JSON export it consumes IS REST). **Both the form actions AND the REST endpoints are now thin adapters over a shared SERVICE LAYER** (ADR-049): `src/lib/server/{tasks,statuses,labels,milestones,views}.ts` hold ALL validation + permission guards + DB writes + side effects (broadcast, notifyMentions, logActivity, recurrence, cascades). A caller parses its input (FormData vs JSON), calls the service (which returns a `ServiceResult<T>` = `{ok:true,data}` | `{ok:false,status,message}`), and maps the result to `fail()`/`json()`. Do NOT re-implement domain logic in a new action or endpoint — call the service and add a thin adapter. Shared helpers: `src/lib/server/graph.ts` (`createsCycle`), `src/lib/server/api.ts` (`parseDateField` + validators). Custom-fields/comments logic already lives in `src/lib/server/{customFields,comments}.ts`. Gate EVERY endpoint per ADR-019: inaccessible → 404 (never 403 on read); 403 only for accessible-but-not-editable; admin/owner checks go BEFORE the existence lookup (no existence oracle); per-user resources (keys) are owner-scoped; never return secrets (key hash, `storagePath`). KEEP `static/llms.txt` updated for any REST change.
- Integrations: `integration` table (one row per `(organizationId, type)` — per-org since ADR-062; org required in code on every write), managed at /integrations (org owner/admin of the ACTIVE org). `src/lib/server/integrations/index.ts` exports `dispatchEvent(event)` — events carry a `projectId`, which resolves the org whose integration row fires; call it fire-and-forget (`void dispatchEvent(...)`, never await) after mutations; it never throws. Events: `project.created`, `task.created`, `task.completed`. Slack uses an incoming webhook (URL must start with `https://hooks.slack.com/`; never sent back to the client). New integration = new module in `src/lib/server/integrations/` + a case in `dispatchEvent`.
- Realtime (ADR: Realtime): WebSockets at `/ws` for live updates + presence. Server transport is **plain ESM** `src/lib/server/realtime/attach.js` (no `$lib`/TS), shared by the Vite dev/preview plugin (`vite.config.ts`) and prod `server.js` (`npm start` runs it, not `build/index.js`). Auth on upgrade + per-project `subscribe` are checked by forwarding cookies to `GET /api/me` and `GET /api/projects/{id}` (404 = no access, ADR: Auth/access) — the WS layer imports no app modules. Those fetches target the server's OWN bound address (`httpServer.address()`, IPv6-bracketed, loopback for a wildcard bind) — NOT the client `Host` header (that was an SSRF/cookie-leak vector); hardcoding `127.0.0.1` is wrong because the server may be bound IPv6-only (Vite dev binds `::1`, which refuses IPv4 → WS auth 401s). Live connections live on `globalThis.__basketsRealtime` (shared across dev module realms); app mutations call `broadcastProjectChange(projectId, actorId)` (from `src/lib/server/realtime/hub.ts`, fire-and-forget, never throws) after EVERY project form action + task/project REST write. Sync = invalidate-and-refetch: clients debounce `invalidateAll()` on a `changed` ping (reuses all `load()` permission filters; no deltas). Browser client + presence store: `src/lib/realtime.svelte.ts` (reconnect + 30s heartbeat); presence avatars render in the project header.
- Side pane (ADR: Project page): `src/lib/components/SidePane.svelte` is the ONE reusable right-pane chrome (slide-in, Escape-close, scrolling body, `title` or `header` snippet + × ). Task editing (`TaskPanel`), the view **Customize** pane, and the **Milestones** pane all render inside it. Task edit opens the pane on title click in ALL views (Table/List/Board) — no inline editing. **Sub-tasks open their own pane too**: clicking a sub-task title (a table sub-row, or a row in the pane's sub-task table) opens it via the view's `onSelectTask` callback (views' `selected` no longer filters out `parentId`); the sub-task's pane hides the Sub-tasks section (depth-1, no children). **Back link = nav origin, not just data parent**: every in-pane task→task jump (sub-task, `task`-cf link, mention chip, dependency) pushes a **per-view nav-history stack** (`backStack` + `navTask`/`navBack` in each view); the pane shows a `← <origin>` breadcrumb (TaskPanel `back`/`onBack` props) that pops the stack, **falling back to the data `parent`** when the stack is empty (so a directly-opened `?task=` sub-task still shows its parent). The stack resets on any fresh open from OUTSIDE the pane (row/card click, `?task=` URL, close); the origin + the open task both resolve from **`allTasks`** (full project list), so a filtered-out origin/target still renders (the `selected` derived also uses `allTasks` — a filtered-out task no longer just closes the pane). It is an **in-flow flex sibling of `.content`** (`position: relative; flex: 0 0 var(--pane-w)`), portaled (`use:portal`) up into the app shell's `[data-pane-host]` (the `.main-body` flex row in `(app)/+layout.svelte`). The shell is fixed-height (`.shell { height: 100dvh }`, `.main`/`.main-body` `min-height:0`) and `.content` is `overflow: scroll`, so content owns its scrollbars and the pane sits beside it. Opening the pane resizes `.content` but **not its children**: SidePane mirrors its live width to `--pane-w` on the portal host (`.main-body`), and `.content > *` gets `min-width: calc(100% + var(--pane-w,0px))`, so children keep their pre-open width and `.content` scrolls horizontally instead of reflowing (the 900px overlay breakpoint resets this to `100%`). Pane top aligns to the topbar bottom (≈49px), left border, no shadow. **Resizable** via a left-edge drag handle (sets `--pane-w` on the aside too, clamps 320…min(900,92vw), persists in `localStorage('sidePaneWidth')`). Under 900px = fixed full-width overlay, handle hidden. (`Popover.svelte` panels **portal to `<body>`** with fixed positioning — flip + viewport clamp, reposition on scroll/resize — so they no longer clip at the pane boundary; outside-click checks both the trigger and the portaled panel. Popover also closes on **Escape in the CAPTURE phase + `stopPropagation`** so Escape dismisses the popover and never bubbles to the SidePane's Escape-closes-pane handler — ADR-055. `SidePane`'s Escape handler additionally no-ops when the target is a native date/time input or a confirm dialog is open.) Only ONE pane is open at a time — `src/lib/sidePaneRegistry.ts` (`openPane`/`closePane`, called from SidePane's onMount/onDestroy) closes the previously-open pane when another opens, across all owners (task pane + Customize + Milestones). **Panes are URL-addressable (ADR-054)**: the open pane is mirrored to the URL via **SvelteKit shallow routing** (no `load()` re-run) so it's copy-shareable + restorable in another window. The **task pane owns `?task=<id>`** — each of the 6 views pairs its existing read-effect (URL→`selectedId`) with a **write-effect** (`selectedId`→URL; FlowView gained the deep-link it lacked); the **project page owns `?pane=customize|milestones|new-task|new-milestone`** (its own read/write-effect pair + init-from-URL; the two Customize `goto('?view=…')` calls carry `&pane=customize`). Both go through **`src/lib/paneUrl.ts` `setPaneUrl(changes)`**, which bases the new URL on **`window.location.href`, NOT `page.url`** — SvelteKit's shallow `replaceState` updates the address bar but does NOT refresh `$app/state`'s `page.url`, so building on it reads a stale base and a close-after-open would dedup wrong + no-op. **Symmetrically, the read-effects read the param via `readPaneParam(key)`** (also in `paneUrl.ts`): it _touches_ `page.url` (so real back/forward still re-trigger) but _returns_ the value from `window.location`. Reading `page.url.searchParams` directly would let an `invalidateAll()` (ANY pane-input blur auto-save, or a realtime ping) churn `page`, re-run the effect with the stale (shallow-missing) URL, clear `selectedId`, and close the pane — the "clicking a pane input then clicking out closes the pane" bug (ADR-055). The read↔write effects share a `lastTaskParam`/`lastPaneParam` sentinel that MUST be read via **`untrack(() => …)`** in BOTH (Svelte 5 `$effect` tracks every reactive read, so a _tracked_ sentinel makes one effect re-run the other with a stale `page.url` and clobber state). The registry keeps `?task=`/`?pane=` mutually exclusive (opening one calls the other's `onClose`, which clears its param). Deep-linking either param opens page+pane on load; `replaceState` (not `pushState`) is used so in-pane task→task nav doesn't fill history (browser Back leaves the page, implicitly closing the pane).
- Tasks: one level of nesting only — `task.parentId` set ⇒ sub-task; sub-tasks cannot have children (enforced in createTask AND patchTask's re-parent branch). `patchTask` accepts **`parentId`** to nest an existing task under another (childless + same-project + parent-is-top-level checks), or `null`/empty to **detach a sub-task back to top-level**; REST `PATCH /api/tasks/{id}` accepts the same `parentId` (same guards). Setting a parent to a `done`-category status cascades to its subs (cascade fires on the parent's status-change only — re-parenting under an already-done parent does NOT force the moved task done, matching `createTask`). Task pane (`TaskPanel` in `SidePane`, ADR: Project page): full-width Title + Description (auto-save on blur), a **pills row** between them — Priority/Assignee/Milestone/Order/Location/Blocked-by/Due-date (+ a **Parent** pill shown on sub-tasks only — a `patchTask` `parentId` Popover to move under another top-level task or detach to top-level) — each a `Popover.svelte` (pill + click panel; Milestone/Location panels have search + inline create; the Start-date/Due-date pills open a **custom calendar** `DatePicker.svelte` — a one-click month grid [prev/next · Today · Clear], NOT a native `<input type=date>` mm/dd/yyyy field (ADR-056; date writes go through a small `setDate(taskId,field,value)` fetch helper, not a form). Pills + title/desc persist via the **`patchTask`** action (patches only fields present in the form via `form.has`, so one field never clobbers another). **Description is its own collapsible section** (chevron + "Description", default EXPANDED, persisted per task under `__description` via the `cfCollapse` mechanism, ADR-056) with a **borderless/document-like editor** (`.desc-editor` — no border/shadow/ring, faint hover-focus bg, mirrors the Overview page). Its form uses `enhance … update({ reset: false })` — a contenteditable's value comes from child text, so the default `form.reset()` would wipe it to an empty defaultValue on a client-rendered pane. `createMilestone`/`createLocation` accept an optional `taskId` to create-and-assign atomically. The **Sub-tasks section sits directly below Description** (above custom fields / attachments / labels) and is a **collapsible section** — a header (`.sub-header`) with the chevron+label+count toggle (`.sub-toggle`) on the left and an **icon-only "+" add button pinned far-right** (visible even while collapsed). It **defaults COLLAPSED** (`loadCollapsed(task.id, '__subtasks', true)`) and **auto-expands + persists that the first time a sub-task is added** (0→≥1, via a `subs.length` tracking effect); persisted per task via the same `cfCollapse` mechanism (ADR-045/056). **Sub-tasks render as stacked blocks** (`.sub-list`, only when the open task is a parent — `!task.parentId`): each sub-task = a head row (hover-revealed selection checkbox + clickable title that opens it + **hover-revealed `.sub-actions`**: **Make into task** [`arrow-up-right-square` → `patchTask` `parentId=''`, detaches to top-level] + **Delete** [`trash`, confirm modal] — no hand-typed × glyph) with its editable property pills (Status + Priority/Assignee/Milestone/Due, each a `patchTask` Popover on that sub-task) **stacked underneath** the title (indented past the status). Property pills are **always shown** (status + Priority/Assignee/Milestone/Due, placeholders included) — they used to hide empties until row-hover, but revealing them reflowed the row (the pills wrapped to extra lines at narrow pane widths, shoving siblings), so the hover-toggle was dropped for a stable layout. The sub-task **bulk Move** popover also offers **Make into tasks (no parent)** (`bulkPatchTasks` with `parentId=''`). The sub-task bulk toolbar also has a **Labels** popover (labels are multi-value, so it's a per-label toggle over the whole selection: each row shows `.opt--on` when ALL selected sub-tasks already carry that label; clicking removes it from all, else adds it to all — `bulkSetLabel` action with `labelId` + `add`; keeps the selection so several can be toggled in a row). REST parity: `PATCH /api/tasks/bulk` `set.addLabelIds`/`removeLabelIds`. In the main TableView, expanded sub-rows now render the **same columns as parent rows** (status/title-with-chevron-spacer/optional cols/cf cols) so a sub-task's custom-field values are visible + column-aligned in the project view (a cf cell shows the value only when the field `appliesTo` sub-tasks). The **"+ Add sub-task"** affordance is the header's icon-only "+" — a `Popover` search (same pattern as Milestone/Location): it lists childless top-level tasks (re-parent via `patchTask` `parentId`) and, on no match, a **Create task "&lt;term&gt;"** button (new sub-task via `createTask` `parentId`). Its popover panel portals to `<body>` (outside the pane subtree — e2e locators must be page-scoped, not pane-scoped). The pane footer (`.pane-footer`) has **Delete task** (confirm modal) + **Close** (`onClose`).
- Locations (ADR: Domain model): project-scoped `location` table (`title` required; `address`, `latitude`, `longitude` all optional/nullable — a location can be a plain named place with no coords; map plots only those with both lat+lng). The task Location/Milestone popovers show a **Create "&lt;term&gt;"** button only when the search has text and no match (no verbose inline form). `task.locationId` FK (`set null` on delete). Managed (CRUD) in project settings; picked/created from the task Location pill. Map views plot from the picked location's coords, falling back to the legacy freeform `task.location` text (kept for old data, no longer editable in the pane). Create/edit/delete need `canEditProject`; _picking_ needs only `canEditTask`.
- Custom fields (ADR: Domain model): project-scoped typed fields on **tasks OR the project itself**. `custom_field.entity` = `task` (default; values in `task_custom_value`) | `project` (values in `project_custom_value`, composite PK `(projectId,fieldId)`). The /projects/[id]/settings Custom fields editor has **[Task | Project] tabs** (`CustomFieldEditor` filters its list + sets `entity` on create); project-entity fields' VALUES are edited in a "Project fields" section there via `CustomFieldValue` (`formAction="?/patchProjectCustomValues"` → `writeProjectCustomValues`, the project-side mirror of `writeTaskCustomValues`). Guard EVERY project/task read with `canAccessProject` (return 404, not 403 — inaccessible ≡ missing) and filter every list (`accessibleWorkspaceIds(user, orgId)` + `grantedProjectIds(user, orgId)` — NO 'all' sentinel; both return concrete Sets). Task editing needs project ACCESS (`canEditTask` ⇒ `canAccessProject`). Structure edits (workspace meta, project CREATE-in-workspace/meta/delete, views, eligible statuses, milestones, project labels, org integrations) need org owner/admin, workspace owner, or a `permission` grant row — check via `canEditWorkspace/Project/View` in EVERY structure mutation (form action + REST). Grant writes: grantee MUST be a member of the resource's org (identical error for nonexistent vs out-of-org id — no oracle) and stamp `permission.organizationId` (grant org == resource org); grants are purged on member removal + re-join (`purgeStaleGrants`). Cascade: org owner/admin ⇒ its workspaces ⇒ projects ⇒ views. View grants do NOT confer visibility. **Rosters: the ORG is the roster boundary** (any member can list org members — the plugin's listMembers is live HTTP anyway) **but per-project pickers stay narrower:** `projectAccessUserIds(projectId, workspaceId)` (org owners/admins + workspace owner + workspace/project grantees — NOT instance admins) ∪ already-referenced users (task assignees, person-cf values). Assignee writes validate against that roster. Grant pickers in settings offer org members only (instance admins included get the same org-scoped roster). There is no "list all users" REST endpoint. **`/projects` (and the sidebar) show only the CURRENT org's current workspace's projects** — `org` cookie then `workspace` cookie (both validated, fallback = first accessible); switching org clears the workspace cookie; project-page loads auto-align the org cookie. Access-filtering alone is not enough — also scope lists to the current org + workspace.
- Start / due dates: `task.startDate` (Start-date pill in `TaskPanel`, next to the Due-date pill — both `align="right"` to avoid overflowing the pane), `milestone.startDate` (NewMilestonePane + settings create + Milestones-pane edit, alongside `targetDate`), and `project.startDate` + `project.dueDate` (project settings → General, side by side) — all optional `<input type=date>`s persisted via the same patch/update actions as their existing date fields.
- Milestones (`milestone`, per project, `task.milestoneId` same-project; cols incl. `description`, `startDate`/`targetDate`, `position`) — managed by the shared **`MilestonesManager.svelte`** rendered in BOTH the /projects/[id]/milestones settings page AND the project "…" → Milestones… pane. Each milestone is a **card**: inline-editable name + **description** (textarea, `update({reset:false})`), a **progress bar** (done/total top-level tasks, green at 100%), an **Overdue** badge (targetDate past + <100%), start/target date pickers, a **Depends on** popover (`milestone_dependency`, DFS cycle-checked), trash-delete (confirm), and a **drag handle** to reorder (pointer-based `use:sortable`, touch-capable → `reorderMilestone` action → `reorderMilestones` in `src/lib/server/milestones.ts`; REST `PATCH /api/projects/{id}/milestones` `{order}`). The component posts to `?/updateMilestone`/`?/deleteMilestone`/`?/createMilestone`/`?/setMilestoneDeps`/`?/reorderMilestone` — all present on BOTH routes (settings + project page). Progress is computed by `milestoneProgressByProject` (settings load) or a client map from `data.tasks` (project pane). `canEdit=false` renders read-only. Labels (`label`/`label_group` — workspace-scoped, managed in workspace settings, attached via `project_label`/`task_label` joins; pickers offer only the project's workspace's labels; each label has an optional `color` (hex) + `icon` (emoji/`iconoir:`) set via color swatch + `IconPicker`, rendered as a tinted `LabelChip.svelte` in TaskPanel/Table/List; a label is EITHER workspace-scoped (`label.workspaceId`, shared pool, toggled onto projects via `project_label`) OR **project-scoped** (`label.projectId` — owned by one project, created/edited/deleted in that project's settings Labels section, always available to its tasks; the project view load unions workspace + project-scoped labels), dependencies (`project_dependency`, `task_dependency`, `milestone_dependency` — same project, sub-tasks only on siblings, DFS cycle check). All dependencies are informational only.
- **Essential features (ADR-032)** — features added schema-first (`task.startDate/recurrence/coverFileId`, tables `comment`/`activity`/`notification`/`template`): **search + filter** (`FilterBar.svelte`, pure `taskFilter.ts`/`taskSort.ts`, set lives in `view.config.filters`; views get a pre-filtered `tasks` prop) — facets are **inclusion sets** (ADR-035, supersedes ADR-033): each `filters` array holds the CHECKED (shown) values and a facet filters ONLY when its key is **present** — absent key = inactive (all shown, all options render checked; new views have no `filters` → all checked). `matchTask` keeps a task only if its facet value is in the list (labels: kept if ANY of its labels is checked; `_none` shows unlabeled/unassigned/no-milestone/no-due; empty present array shows nothing). FilterBar/Customize `toggle` seeds the full set on first uncheck + deletes the key when all re-checked; badge = count of unchecked. NOTE: pre-ADR-035 stored filters read backwards until cleared. The FilterBar facet pills live behind an **icon-only `filter-list` button** (right of the search box; lightweight toggle dropdown, NOT the shared `Popover` which would clip the nested facet popovers — badge = active-facet count). Free-text **search also matches custom-field values** (resolved to display labels): the page builds a per-task `cfSearchByTask` map and passes it as `FilterHelpers.searchableText`, folded into `matchTask`'s haystack — **comments + activity** (`server/comments.ts` + `TaskComments.svelte`; `logActivity` from create/patch/setStatus); **notifications** (`server/notifications.ts` + `NotificationBell.svelte`); **timeline** + **calendar** view types (`TimelineView`/`CalendarView.svelte`; also add new types to the server `VIEW_TYPES` allow-list in `server/projects.ts`); **bulk actions** (`selection.svelte.ts` + `BulkActionBar.svelte`; `bulkPatchTasks`/`bulkDeleteTasks`/`bulkReparentToNew`/`bulkSetLabel`; reused for **sub-task bulk edit** in the task pane, incl. a multi-value **Labels** toggle); **templates + recurring** (`server/templates.ts`, `recurrence.ts`); **CSV export** (`/api/projects/{id}/export`); **photo capture** (`TaskAttachments.svelte`, `/api/tasks/{id}/files`); **mobile** (`mobile.css` — global rules keyed on LITERAL class names inside `max-width` media queries; besides the drawer/viewbar/table/pane rules it also makes the **custom-fields + project-settings editors touch-friendly**: forces hover-only `.icon-btn` row actions visible + ~40px on touch [no hover on phones, else edit/delete were unreachable], wraps `.cf-row`, stacks `.cf-row--edit`/`.cf-create`/`.grant-form` controls + the settings `.date-row`/`.grant-row` full-width, and bumps tiny tap targets `.cp-swatch`/`.select--mini`/`.x-btn`). New REST endpoints are documented in `static/llms.txt`. `Popover.svelte` has an `up` prop for bottom-anchored bars.

## Conventions

- **Icons (ADR-052, was Cross-cutting)**: use **Heroicons** (24px outline) via `Icon.svelte` (`<Icon name="chevron-down" size={14} />`) for UI chrome — glyphs come from the external sprite `static/heroicons.svg`, built by `scripts/build-heroicons-sprite.mjs` (`npm run icons:build`, also a prebuild step) from the `heroicons` dep. The build ALSO emits **alias symbols** for the ~60 legacy **iconoir tokens** the code still hardcodes (`nav-arrow-down`→`chevron-down`, `task-list`→`queue-list`, `xmark`→`x-mark`, `edit-pencil`→`pencil-square`, `trash`, `plus`, …) so existing `<Icon name="…">` call sites render Heroicons unchanged; two glyphs iconoir had but Heroicons lacks (`circle`, `git-commit`) are hand-authored custom symbols. A build-time assertion fails if any alias target is missing. **Two generated name lists** in `src/lib/heroiconNames.ts`: `HEROICON_NAMES` (~324, the `IconPicker` search pool) + `ICON_NAMES` (all 367 sprite ids incl. aliases — the set `parseIconValue`/project-icon endpoints validate a stored `iconoir:<name>` against). Don't hand-write unicode glyph chars for affordances (…, ▸, ×, chevrons, view-type icons). Stored entity icon values render via `EntityIcon`; they keep the historical **`iconoir:` prefix** as the sprite-icon namespace (a picked Heroicon is stored `iconoir:<heroicon-name>`; the prefix is a name, not a library). Keep semantic custom glyphs (PriorityIcon bars, BoardView status dots).
- **Color picker (ADR-040)**: pick colors with **`src/lib/components/ColorPicker.svelte`** — the color twin of `IconPicker` (280px panel-in-`Popover`, controlled `{ value, onSelect(hex), onRemove? }`, caller owns persistence). Shows a 21-swatch `COLOR_PRESETS` grid (Tailwind-500/600; the 5 status defaults are members) + a custom-hex input + a styled native `<input type=color>` "Custom" hatch. `normalizeHex` emits lowercase **6-digit** `#rrggbb` (server `parseColor` is 6-digit-only). a11y: `role=radiogroup` of `role=radio` swatches, roving tabindex, `aria-checked` (NOT `aria-pressed`). Used at the 4 color sites (StatusEditor, CustomFieldEditor, project labels, workspace labels) via a `Popover` with a `.cp-swatch` trigger — form sites add a hidden `<input name="color">`; label-edit sites call the `patchLabel`/`patchProjectLabel` fetch helper from `onSelect`. Don't reintroduce raw `<input type="color">`/`.color-in`.
- **Tooltips (ADR-034)**: `use:tooltip={text | { text, placement, delay }}` from `src/lib/tooltip.ts` — a Svelte action backed by ONE shared floating node (themed via `.app-tooltip` in `app.css`, inverse bubble, ≤120ms fade, reduced-motion aware). Shows on hover (mouse only) + keyboard focus after a delay, flips at the viewport edge, hides on leave/click/Escape/scroll, and **strips the host's native `title`** (no OS double). Use it for icon-only affordances (buttons, drag/resize handles, avatars, badges, relative timestamps) instead of native `title=`. (`IconPicker`'s large emoji/icon grid keeps native `title` for perf.)
- **No decorative glyphs** (ADR: Project page): don't prefix entity names with ornaments (`◇` milestone, `●` label dot, `⛓` project) — show the plain name. (A `⛓ N` blocker _count_ badge is fine; it's a count, not a name prefix.) Statuses keep their color (the pill tint / swatch carries data, not decoration).
- **i18n**: every user-facing UI string goes through `$t('English text')` from `$lib/i18n` — English strings are the keys (en = passthrough). App is English-only (Vietnamese removed); the `$t` wrapper + `registerDictionary` are kept so a locale can be re-added by registering a dictionary. Never translate DB content (status/label/project/task/user names).
- **llms.txt**: `static/llms.txt` documents the REST API for agents (served at `/llms.txt`). KEEP IT UPDATED whenever a REST endpoint, request/response field, auth, or status/workspace rule changes.

- **ADR.md**: every architecture/design decision (new dependency, schema approach, auth model, API shape, theming strategy, etc.) gets a new `ADR-NNN` record — context, decision, consequences. Update an existing record's status (`Superseded by ADR-NNN`) instead of rewriting history.

- **Theming (ADR: Cross-cutting)**: Tailwind 4 + DaisyUI 5. `src/app.css` = `@import "tailwindcss"; @plugin "daisyui" { themes: light --default, dark --prefersdark; }` + fonts (Inter/IBM Plex Mono) + a **token bridge** (legacy `--color-*`/`--sp-*`/`--dur-*`/`--radius` vars mapped onto DaisyUI theme tokens like `--color-base-100`/`--color-base-content`/`--color-base-300`, so component-scoped `<style>` blocks follow the active theme automatically). Prefer DaisyUI component classes (`btn btn-primary`, `input`, `select`, `card`, `badge badge-neutral`, `alert alert-error`, `menu`, `modal`, `dropdown`) and Tailwind utilities; don't hardcode colors — use DaisyUI tokens or the bridged `var(--color-*)`. `.card`, `.label`, `.field`, `.u-*` are thin app-specific helpers kept in app.css. Focus rings are removed app-wide (`:focus, :focus-visible { outline: none }` in app.css). Theme = `theme` cookie, applied to `<html data-theme>` at SSR via `hooks.server.ts` `transformPageChunk`; toggled from the topbar **Appearance** dropdown in `(app)/+layout.svelte` (a `half-moon` button → `role=menu` with **Dark mode** + **High contrast** `menuitemcheckbox` rows; `theme`/`contrast` seeded from root `+layout.server.ts` load). Add a theme = add it to the `themes:` list (and the `THEMES` arrays in hooks + root layout load). **High contrast (ADR-038)** is a SEPARATE axis: a `contrast` cookie → `<html data-contrast="high">` at SSR (same `transformPageChunk`), toggled from the same topbar **Appearance** menu AND the checkbox in /settings → **Theming**; `[data-contrast='high']` in app.css overrides the bridge + DaisyUI tokens to black-on-white (incl. forcing DaisyUI's faint `.table` cell borders solid). Data-driven accents (status/label colors) keep their inline colors.
- Tabs for indentation; single quotes. Now **mechanically enforced** via ESLint + Prettier + EditorConfig (ADR-059): `.prettierrc` (tabs, single quotes, no trailing comma, printWidth 100), flat `eslint.config.js` (permissive — noisy rules `warn`, real bugs `error`; `npm run lint` must stay 0-error). Run `npm run format` before committing bulk edits; don't hand-fight the formatter.
- Single instance, multi-tenant (ADR-062): one deployment/DB hosts many organizations with row-level isolation enforced in code (org membership first, always). Workspace ownership/grants remain the intra-org sharing primitive. Invitations are link-as-capability (`/invite/<id>`, no SMTP). Multi-tenant PRODUCTION deploys should set `DB_DIALECT=postgres` (sqlite's `withTransaction` is a single shared connection).
- Destructive confirms: never `onsubmit` + `preventDefault()` on a `use:enhance` form (enhance's fetch fires anyway) — use the enhance callback's `cancel()`, OR the centered confirm modal. **Confirm modal**: `confirmDialog(message, { confirmLabel?, danger? })` from `src/lib/confirm.svelte.ts` returns a `Promise<boolean>`; `src/lib/components/ConfirmModal.svelte` (mounted once in the `(app)` shell, like Toaster) renders a page-centered dialog (fixed backdrop, Escape/backdrop = cancel, Enter = confirm). **Escape/Enter are handled in the CAPTURE phase + `stopPropagation`** so a cancel-Escape can't also close a SidePane open behind the dialog, and **Enter only confirms when focus is inside the dialog** (a background input's Enter can't trigger a danger delete) — ADR-055. Pattern for an enhance form: make the trigger a `type="button"`, `await confirmDialog(...)`, then `form?.requestSubmit()` (don't await inside enhance — its fetch already fired). Task + sub-task deletes (`TaskPanel`) use this (danger, "Delete this task[ and its sub-tasks]?" / "Delete this sub-task?").
- Motion: no animation library (ADR: Cross-cutting). Use `svelte/transition`/`animate:flip` with the `--dur-fast/--dur/--dur-slow` tokens; everything ≤200ms. **Page swap**: `(app)/+layout.svelte` wraps `.content` in `{#key page.url.pathname}` and re-plays a subtle CSS `@keyframes pageFade` (opacity .5→1, ~140ms) on every navigation — a barely-there fade, NOT a full blank (replaces an earlier jarring `in:fade`). **List cascade**: `.stagger-in` (in `app.css`) makes a container's direct children fade+rise in sequence via `sibling-index()` (with an `@supports` fallback → all-at-once where unsupported, e.g. Firefox; capped at 400ms); used on the projects grid. **Scroll**: persistent chrome regions whose vertical scrollbar toggles (sidebar, side-pane body) use `scrollbar-gutter: stable` so content doesn't shift when the bar appears. The global `@media (prefers-reduced-motion: reduce)` rule (app.css) collapses all of the above.
- **Touch / mobile (ADR-042)**: drag-and-drop + right-click must work on touch — native HTML5 DnD (`draggable`/`ondragstart`/`ondrop`) and `oncontextmenu` are mouse-only. Use the in-house **pointer-event** primitives instead: **`use:sortable`** (`src/lib/sortable.ts`) for list reorder — container action, items get `data-sortable-id`, drag from an optional `[data-sortable-handle]`; mouse drags on a move threshold, touch engages after a still HOLD (so a swipe still scrolls), calls `onReorder(ids)`; and **`use:longpress`** (`src/lib/longpress.ts`) which fires the SAME handler as the mouse `oncontextmenu` (keep both on the element). Both swallow the trailing click. **BoardView** card drag is bespoke pointer-based (hold-to-engage; `document.elementFromPoint` hit-tests `data-lane`/`data-status`/`data-index`; the dragged card gets `pointer-events:none`). Add `-webkit-touch-callout:none` to long-pressable/draggable elements. Adopted: view-tab menu (mouse still drag-reorders tabs; touch tab-reorder is out of scope), BoardView col-head menu + card drag, StatusEditor (one sortable per category), CustomFieldEditor, chip-fields Show bar (tap to add/remove, handle-drag to reorder). Don't reintroduce native `draggable` for new reorder UI; resize handles already use pointer events. **`:hover`-revealed controls on a TAP TARGET cause an iOS "2-tap" bug** — a tap whose first effect is a hover-gated appearance (a checkbox fading in, a title underline) is treated by iOS Safari as "reveal hover" and the click is swallowed, so opening a task took two taps. `src/lib/mobile.css` neutralizes this under `@media (hover: none)` by forcing those hovered states on (`.bulk-check`/`.col-select input`/`.group-add` opacity, `.task-title`/`.row-title` underline) with `!important` (to beat the scoped component `opacity:0`). Any new hover-revealed affordance on a clickable row needs the same neutralization. **Reorder-form flush gotcha**: the sortable `onReorder` pattern sets a `$state` `reorderIds` bound to a hidden `<input>` then submits — call **`await tick()` before `reorderForm.requestSubmit()`** (Svelte 5 batches the state→DOM write, so a synchronous submit posts the STALE/empty value and the server rejects it as "Invalid order"). Fixed in `StatusEditor`/`CustomFieldEditor`. **Mobile UI pass (ADR-053)**: (1) `src/app.html` sets **`viewport-fit=cover`** so `env(safe-area-inset-*)` is non-zero on notched phones — the topbar padding, drawer/backdrop `top` (mirror the topbar's `max(--sp-2, env-top)`, don't add the inset twice), the fixed bottom bars (`.toaster`/`.bulk-bar`) and the task-sheet `.pane-body`/`.pane-footer` are all wired to the insets. (2) `app.css` `body` has `-webkit-text-size-adjust:100%` + transparent tap-highlight, and `mobile.css` forces **`font-size:16px` on inputs/`select`/`textarea` under `@media (hover:none)`** to stop iOS zoom-on-focus (any new text field <16px will zoom — keep it ≥16 or rely on this rule). (3) The **full-screen task sheet** covers the topbar on phones: SidePane's own `@media (max-width:900px)` overlay rule sets **`z-index:40`** (> the topbar's `10`) so its header + × are reachable (a scoped rule out-specifies `mobile.css`, so set it there, not in `mobile.css`). (4) The project "…" menu's **hover fly-outs become click-toggled inline accordions on mobile** — an `openSub` state + `.menu-sub.open`, with `:hover`/`:focus-within` gated behind `@media (hover:hover)` so touch has one non-fighting driver (hover has no touch equivalent; the same applies to any new hover-opened submenu). (5) Per-view density: board horizontal-scroll columns, calendar/dashboard/timeline sizing, map/flow mobile heights + larger Leaflet/Flow controls (MiniMap hidden), MentionEditor menu clamps to `visualViewport.height` (above the keyboard).
- `.env` is gitignored; secrets read via `$env/dynamic/private` (server) or `process.env` (scripts).
