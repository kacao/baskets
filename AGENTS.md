# CLAUDE.md

Baskets — single-tenant project management app (projects → tasks → sub-tasks). Minimalist. UI built on Tailwind 4 + DaisyUI 5 (ADR: Cross-cutting). Product scope: `PRD.md`. Architecture decisions: `ADR.md`.

## Commands

```bash
npm run dev          # dev server on :5173
npm run build        # production build (adapter-node → build/)
npm run check        # svelte-kit sync + svelte-check (run before committing)
npm run db:push      # apply src/lib/server/db/schema.ts to SQLite (drizzle-kit push)
npm run db:seed      # seed admin/demo users + sample data (idempotent; skips if admin exists)
npm run start        # serve production build via custom server.js (adapter-node handler + /ws WebSocket; reads .env)
```

No test suite yet. Verify changes with `npm run check` + manual smoke test.

db:push gotcha: when one push both rebuilds a table (e.g. dropping a unique) AND adds a column to it, drizzle-kit generates a broken `INSERT … SELECT new_column FROM old_table` and dies mid-push, leaving the DB partially migrated. Back up the .db first; if it happens, apply the missing `ALTER TABLE … ADD COLUMN` by hand, then re-run `npx drizzle-kit push --force` until "Changes applied".

## Stack

- SvelteKit 2 + **Svelte 5 runes** (`$props`, `$state`, `$derived` — no legacy `export let` / `$:`)
- UI: Tailwind 4 (`@tailwindcss/vite`) + DaisyUI 5 (see Theming under Conventions)
- SQLite via better-sqlite3 + Drizzle ORM (`DATABASE_URL` in `.env`, default `./data/baskets.db` — DB lives in the gitignored `data/` dir, which is kept in git via `data/.gitkeep`)
- BetterAuth 1.6 with `twoFactor` (TOTP) and `admin` plugins

## Architecture

- `src/lib/server/db/schema.ts` — ALL tables: BetterAuth tables (user, session, account, verification, twoFactor) AND app tables (workspace, project, task, …). BetterAuth field requirements change between versions; if it errors about a missing field, add the column here and `npm run db:push`. Keep `scripts/seed.ts`'s auth config in sync with `src/lib/server/auth.ts`.
- Workspaces: every project belongs to one (`project.workspaceId`); workspaces own custom statuses and ALL labels/groups. `workspace.ownerId` + `permission` grants (`resourceType 'workspace'`) gate workspace structure edits and cascade to its projects/views — and per ADR: Auth/access they also define VISIBILITY (non-admins only see workspaces they own/are granted, plus directly granted projects). `ensureDefaultWorkspace` (hooks) bootstraps a "Default" workspace and adopts pre-workspace rows; the nullable `workspaceId` FKs exist only for db:push on populated tables — code treats them as required. Workspace CRUD at /workspaces; per-workspace settings pane (general/statuses/labels/grants/delete) at /workspaces/[id]/settings. A workspace is deletable only when empty; the last one can't be deleted. A user's FIRST workspace (they own none yet — `isFirstWorkspaceForUser`) is seeded with sample projects/milestones/tasks via `seedWorkspaceSamples` (in `projects.ts`). The sidebar header is a workspace switcher (current workspace name + chevron; `workspace` cookie, fallback = first workspace) and the sidebar project list is scoped to the current workspace. **"Projects" is a section label** (not a link); each project is a **collapsible menu** (chevron, auto-expands the active project) → **Tasks / Milestones / Statuses / Labels / Custom fields / Locations / Settings** sub-items with iconoir icons. Each is its own route under `/projects/[id]/*`: the 5 structure sub-pages (`milestones`/`statuses`/`labels`/`custom-fields`/`locations`) **re-export the settings `load`+`actions`** (`ProjectSettingsData` shared type — single source of truth; their forms post `?/action` to their own route), and `/projects/[id]/settings` is trimmed to General/Depends/Grants/Danger + an **Import / Export** section (ADR-039).
- `src/lib/server/auth.ts` — BetterAuth instance. `src/lib/auth-client.ts` — browser client (used for auth + admin actions). `src/hooks.server.ts` — populates `locals.user` / `locals.session` on every request.
- Routes: `(auth)` group = login/register/two-factor (redirects away if signed in); `(app)` group = everything else (redirects to /login if not; its `+layout.svelte` is the shell with sidebar). `/admin` additionally requires `locals.user.role === 'admin'` (enforced in its `+page.server.ts`).
- Data mutations: SvelteKit form actions in `+page.server.ts` files (projects, tasks). Exception: auth/2FA/admin-user actions go through `authClient` (BetterAuth API) client-side.
- REST API (ADR-036 — **full UI→API parity**): `src/routes/api/**/+server.ts`. Auth via session cookie OR `Authorization: Bearer bsk_…` API key. Keys are managed per-user in /settings (SHA-256 hash stored in `api_key` table, plaintext shown once); `src/lib/server/api-keys.ts` generates/resolves them, `src/hooks.server.ts` resolves the bearer header into `locals.user`. Shared validation helpers in `src/lib/server/api.ts`. **Every web-UI form action now has a REST endpoint** (resource-oriented): projects, tasks (+ `bulk`, `labels`, `dependencies`), comments, files, notifications, templates (+ `DELETE /api/templates/{id}`), export, me — PLUS workspaces (+ nested `statuses`/`labels`/`label-groups`/`grants`), `statuses/{id}`, `labels/{id}`, `label-groups/{id}`, project `statuses`/`labels`/`milestones`/`locations`/`views`/`custom-fields`/`custom-values`/`dependencies`/`grants`, `milestones/{id}`(+`/dependencies`), `locations/{id}`, `views/{id}`, `custom-fields/{id}`(+`/options`), `custom-field-options/{id}`, `keys`. EXCLUDED on purpose: **Integrations** (write-only Slack secret, UI-only) and auth/account/2FA (BetterAuth). Endpoints port logic from the form actions (the action is the behavioral source of truth — logic is duplicated, not yet extracted). Gate EVERY endpoint per ADR-019: inaccessible → 404 (never 403 on read); 403 only for accessible-but-not-editable; admin/owner checks go BEFORE the existence lookup (no existence oracle); per-user resources (keys) are owner-scoped; never return secrets (key hash, `storagePath`). KEEP `static/llms.txt` updated for any REST change.
- Integrations: `integration` table (one row per `type`, single tenant), managed at /integrations. `src/lib/server/integrations/index.ts` exports `dispatchEvent(event)` — call it fire-and-forget (`void dispatchEvent(...)`, never await) after mutations; it never throws. Events: `project.created`, `task.created`, `task.completed`. Slack uses an incoming webhook (URL must start with `https://hooks.slack.com/`; never sent back to the client). New integration = new module in `src/lib/server/integrations/` + a case in `dispatchEvent`.
- Realtime (ADR: Realtime): WebSockets at `/ws` for live updates + presence. Server transport is **plain ESM** `src/lib/server/realtime/attach.js` (no `$lib`/TS), shared by the Vite dev/preview plugin (`vite.config.ts`) and prod `server.js` (`npm start` runs it, not `build/index.js`). Auth on upgrade + per-project `subscribe` are checked by forwarding cookies to `GET /api/me` and `GET /api/projects/{id}` (404 = no access, ADR: Auth/access) — the WS layer imports no app modules. Those fetches target the server's OWN bound address (`httpServer.address()`, IPv6-bracketed, loopback for a wildcard bind) — NOT the client `Host` header (that was an SSRF/cookie-leak vector); hardcoding `127.0.0.1` is wrong because the server may be bound IPv6-only (Vite dev binds `::1`, which refuses IPv4 → WS auth 401s). Live connections live on `globalThis.__basketsRealtime` (shared across dev module realms); app mutations call `broadcastProjectChange(projectId, actorId)` (from `src/lib/server/realtime/hub.ts`, fire-and-forget, never throws) after EVERY project form action + task/project REST write. Sync = invalidate-and-refetch: clients debounce `invalidateAll()` on a `changed` ping (reuses all `load()` permission filters; no deltas). Browser client + presence store: `src/lib/realtime.svelte.ts` (reconnect + 30s heartbeat); presence avatars render in the project header.
- Side pane (ADR: Project page): `src/lib/components/SidePane.svelte` is the ONE reusable right-pane chrome (slide-in, Escape-close, scrolling body, `title` or `header` snippet + × ). Task editing (`TaskPanel`), the view **Customize** pane, and the **Milestones** pane all render inside it. Task edit opens the pane on title click in ALL views (Table/List/Board) — no inline editing. **Sub-tasks open their own pane too**: clicking a sub-task title (a table sub-row, or a row in the pane's sub-task table) opens it via the view's `onSelectTask` callback (views' `selected` no longer filters out `parentId`); the sub-task's pane shows a `← <parent>` breadcrumb (`onSelectTask` back to the parent) and hides the Sub-tasks section (depth-1, no children). It is an **in-flow flex sibling of `.content`** (`position: relative; flex: 0 0 var(--pane-w)`), portaled (`use:portal`) up into the app shell's `[data-pane-host]` (the `.main-body` flex row in `(app)/+layout.svelte`). The shell is fixed-height (`.shell { height: 100dvh }`, `.main`/`.main-body` `min-height:0`) and `.content` is `overflow: scroll`, so content owns its scrollbars and the pane sits beside it. Opening the pane resizes `.content` but **not its children**: SidePane mirrors its live width to `--pane-w` on the portal host (`.main-body`), and `.content > *` gets `min-width: calc(100% + var(--pane-w,0px))`, so children keep their pre-open width and `.content` scrolls horizontally instead of reflowing (the 900px overlay breakpoint resets this to `100%`). Pane top aligns to the topbar bottom (≈49px), left border, no shadow. **Resizable** via a left-edge drag handle (sets `--pane-w` on the aside too, clamps 320…min(900,92vw), persists in `localStorage('sidePaneWidth')`). Under 900px = fixed full-width overlay, handle hidden. (`Popover.svelte` panels **portal to `<body>`** with fixed positioning — flip + viewport clamp, reposition on scroll/resize — so they no longer clip at the pane boundary; outside-click checks both the trigger and the portaled panel.) Only ONE pane is open at a time — `src/lib/sidePaneRegistry.ts` (`openPane`/`closePane`, called from SidePane's onMount/onDestroy) closes the previously-open pane when another opens, across all owners (task pane + Customize + Milestones).
- Tasks: one level of nesting only — `task.parentId` set ⇒ sub-task; sub-tasks cannot have children (enforced in createTask AND patchTask's re-parent branch). `patchTask` accepts **`parentId`** to nest an existing task under another (childless + same-project + parent-is-top-level checks), or `null`/empty to **detach a sub-task back to top-level**; REST `PATCH /api/tasks/{id}` accepts the same `parentId` (same guards). Setting a parent to a `done`-category status cascades to its subs (cascade fires on the parent's status-change only — re-parenting under an already-done parent does NOT force the moved task done, matching `createTask`). Task pane (`TaskPanel` in `SidePane`, ADR: Project page): full-width Title + Description (auto-save on blur), a **pills row** between them — Priority/Assignee/Milestone/Order/Location/Blocked-by/Due-date (+ a **Parent** pill shown on sub-tasks only — a `patchTask` `parentId` Popover to move under another top-level task or detach to top-level) — each a `Popover.svelte` (pill + click panel; Milestone/Location panels have search + inline create; the Due-date pill's panel is a `<input type=date>` + Clear). Pills + title/desc persist via the **`patchTask`** action (patches only fields present in the form via `form.has`, so one field never clobbers another). The Description form uses `enhance … update({ reset: false })` — a `<textarea>`'s value comes from child text, so the default `form.reset()` would wipe it to an empty defaultValue on a client-rendered pane. `createMilestone`/`createLocation` accept an optional `taskId` to create-and-assign atomically. **Sub-tasks render as stacked blocks** (`.sub-list`, only when the open task is a parent — `!task.parentId`): each sub-task = a head row (hover-revealed selection checkbox + clickable title that opens it + **hover-revealed `.sub-actions`**: **Make into task** [`arrow-up-right-square` → `patchTask` `parentId=''`, detaches to top-level] + **Delete** [`trash`, confirm modal] — no hand-typed × glyph) with its editable property pills (Status + Priority/Assignee/Milestone/Due, each a `patchTask` Popover on that sub-task) **stacked underneath** the title (indented past the status). Property pills are **always shown** (status + Priority/Assignee/Milestone/Due, placeholders included) — they used to hide empties until row-hover, but revealing them reflowed the row (the pills wrapped to extra lines at narrow pane widths, shoving siblings), so the hover-toggle was dropped for a stable layout. The sub-task **bulk Move** popover also offers **Make into tasks (no parent)** (`bulkPatchTasks` with `parentId=''`). The sub-task bulk toolbar also has a **Labels** popover (labels are multi-value, so it's a per-label toggle over the whole selection: each row shows `.opt--on` when ALL selected sub-tasks already carry that label; clicking removes it from all, else adds it to all — `bulkSetLabel` action with `labelId` + `add`; keeps the selection so several can be toggled in a row). REST parity: `PATCH /api/tasks/bulk` `set.addLabelIds`/`removeLabelIds`. In the main TableView, expanded sub-rows now render the **same columns as parent rows** (status/title-with-chevron-spacer/optional cols/cf cols) so a sub-task's custom-field values are visible + column-aligned in the project view (a cf cell shows the value only when the field `appliesTo` sub-tasks). The **"+ Add sub-task"** affordance is a `Popover` search (same pattern as Milestone/Location): it lists childless top-level tasks (re-parent via `patchTask` `parentId`) and, on no match, a **Create task "&lt;term&gt;"** button (new sub-task via `createTask` `parentId`). The pane footer (`.pane-footer`) has **Delete task** (confirm modal) + **Close** (`onClose`).
- Locations (ADR: Domain model): project-scoped `location` table (`title` required; `address`, `latitude`, `longitude` all optional/nullable — a location can be a plain named place with no coords; map plots only those with both lat+lng). The task Location/Milestone popovers show a **Create "&lt;term&gt;"** button only when the search has text and no match (no verbose inline form). `task.locationId` FK (`set null` on delete). Managed (CRUD) in project settings; picked/created from the task Location pill. Map views plot from the picked location's coords, falling back to the legacy freeform `task.location` text (kept for old data, no longer editable in the pane). Create/edit/delete need `canEditProject`; *picking* needs only `canEditTask`.
- Custom fields (ADR: Domain model): project-scoped typed fields on **tasks OR the project itself**. `custom_field.entity` = `task` (default; values in `task_custom_value`) | `project` (values in `project_custom_value`, composite PK `(projectId,fieldId)`). The /projects/[id]/settings Custom fields editor has **[Task | Project] tabs** (`CustomFieldEditor` filters its list + sets `entity` on create); project-entity fields' VALUES are edited in a "Project fields" section there via `CustomFieldValue` (`formAction="?/patchProjectCustomValues"` → `writeProjectCustomValues`, the project-side mirror of `writeTaskCustomValues`). That section also has a **"Show on project page"** drag-drop bar (two dropzones — Show + Available — native HTML5 DnD, `animate:flip`) to pick + order which project fields render as header chips; it persists the ordered id list via `setProjectChipFields` → `project.chipFields` (see Project page header). Project-entity fields are filtered OUT of the task-facing project-view load + rejected by `writeTaskCustomValues` (entity guard) so they never appear on tasks. Tables: `custom_field` (id/projectId/`entity`/name/`type`/`config` schemaless JSON/`appliesTo` (task entity only)/position — `type` ∈ text|number|select|date|person|files|task|checkbox|email|phone|place|url|**rollup** (rollup = a computed, display-only aggregate: `config.{relation,targetFieldId,formula}` — relation ∈ blocked-by|blocking|sub-task|task (project-entity fields force `task` = all project tasks), formula ∈ count|sum|average|min|max; computed client-side via `computeTaskRollup`/`rollupAggregate`, never stored — `writeTaskCustomValues` ignores it; threaded as a `rollupText` prop into `CustomFieldValue` from TaskPanel/TableView/project-settings); name unique per **(project, `entity`)** in code (a task field and a project field MAY share a name — they're separate value tables; enforced in create + rename, form actions + REST); **type immutable after create**; **`appliesTo`** = `all` (default) | `tasks` (top-level only) | `subtasks`, set in the editor, enforced by `fieldAppliesTo(field, isSubtask)` — filters which fields show in TaskPanel (per the open task's level), NewTaskPane (top-level), and Table columns (`cfCols` = top-level-applicable; sub-rows show a field's cell only when it also applies to sub-tasks; `subtasks`-only fields have no table column and are pane-only); `writeTaskCustomValues` rejects a value whose field doesn't apply to the task's level), `custom_field_option` (select members, enum-with-color/icon like `label`), `task_custom_value` (composite PK `(taskId,fieldId)`, single `value` TEXT — scalar OR a JSON id-array for the multi-capable types select/person/place/files/task (a `task` field with `config.multi` links a LIST of same-project tasks, e.g. an "Inspection" field → many inspection tasks; pre-multi scalar `task` ids are recovered as 1-element arrays by `decodeValue` and the server's lenient parse); absent row = no value), `file` (uploads). Client-safe helpers in `src/lib/customFields.ts` (`CUSTOM_FIELD_TYPES`, `defaultConfig`/`sanitizeConfig`, `decodeValue`, `formatNumber`/`formatDate`), server `src/lib/server/customFields.ts` re-exports + `writeTaskCustomValues` (the ONE validated write path used by `patchTask`/`createTask` form actions AND REST — validates every ref against `field.projectId`) + `customValuesByTask`/`apiCustomFieldEntries` (REST). Field + option CRUD (drag-reorder) lives in /projects/[id]/settings via shared **`CustomFieldEditor.svelte`**; values render+edit through one **`CustomFieldValue.svelte`** with `mode` `pill` (TaskPanel, posts `cf_<fieldId>` to `?/patchTask`), `cell` (optional Table columns keyed `cf:<fieldId>`, toggled like the built-in column show/hide — from the table's `…` menu AND the view **Customize → Columns** section, which lists `cf:<id>` chips for every top-level-applicable field), `input` (NewTaskPane, hidden `cf_<fieldId>` read by `createTask`). In `pill` mode a **multi** reference field (`select`/`person`/`place` with `config.multi`) renders as a wrapping row of **removable chips + an "+ Add" picker** (not one crammed pill); `files` keeps its own list+upload editor. **EXCEPTION: a `task`-type field in the TaskPanel renders as a collapsible sub-task-style list** (`taskCfList` snippet in `TaskPanel.svelte`, `.cf-task`) — each linked task is a block with its title (opens it via `onSelectTask`) + editable Status/Priority/Assignee/Milestone/Due pills (the SAME `sub*` snippets sub-tasks use, `patchTask` on the linked task), a remove (×, unlinks via `cf_<id>`), an "+ Add" search picker, and a chevron header (`cfCollapsed`). Membership writes post `cf_<fieldId>` to `?/patchTask` like `CustomFieldValue`. (CustomFieldValue still renders task chips in `cell`/`input` modes.) v1 is **display-only** (no sort/group/filter by custom fields). Number `custom` format string is stored but NOT interpreted (rendered via `Intl.NumberFormat`; `custom` shows raw). Deferred **formula**/**button** slot in via type+config (no schema change). Threaded through the project page load → views → TaskPanel/NewTaskPane as `customFields`/`customFieldOptions`/`taskCustomValues`/`files`. **`task`-type cf resolution (titles + the link picker) must use the FULL project task list, never the view's filtered one** — else a linked task hidden by a filter renders as a raw UUID. Views take an **`allTasks`** prop (full `data.tasks`, defaults to `tasks`) and pass it to `CustomFieldValue` cells + TaskPanel (which also needs the full list for parent/subs/deps/rollup, incl. TableView's `rollupText`). Free-text search (FilterBar) AND the `task`-cf link picker (`CustomFieldValue` `matchTasks`) also match **custom-field values resolved to display labels**: `buildTaskCfSearch(fields, values, resolvers)` (in `customFields.ts`) builds a per-task `Map<id,string>`, threaded as `FilterHelpers.searchableText` (page) and a `taskSearch` prop (page→NewTaskPane, TaskPanel→CustomFieldValue; TaskPanel builds its own map from its props).
- Files (ADR: Domain model): `files`-type custom-field uploads on **local disk** under `UPLOADS_DIR` (default `./data/uploads/`, created on demand, gitignored — see `src/lib/server/uploads.ts`). Served ONLY via access-gated `GET /api/files/[id]` (streams; `canAccessProject` → 404; `nosniff` + CSP `sandbox`, non-raster-image as `attachment` — no stored-XSS from our origin). `POST /api/files` (multipart: `fieldId` + `file` [+ `taskId`, validated same-project]; 10 MB cap; denylist of exec + active-markup exts (svg/html/xml/js/…); **MIME server-derived from extension** via `mimeForExt`, NOT the client claim; project derived from the field), `DELETE /api/files/[id]` (unlinks row + disk + strips id from referencing values). `storagePath` is never sent to the client. Deleting a files-field unlinks its bytes first (`deleteFilesForField`).
- Statuses: table-driven (`status` + `project_status` eligibility). Behavior keys off `status.category` (`backlog|planned|in-progress|completed|canceled` — one per default status; `completed` = the "done" bucket that completes sub-tasks; `backlog` = new-task default), never the name. Categories + display labels are in the client-safe `src/lib/statuses.ts` (`STATUS_CATEGORIES`/`categoryLabel`/`DONE_CATEGORY`/`INITIAL_CATEGORY`), re-exported by `src/lib/server/statuses.ts`; `ensureDefaultStatuses` reconciles built-in name/category/color on boot. Three scopes: app-wide = exactly the five built-ins (Backlog/Planned/In progress/Completed/Canceled), FIXED name/category/no-add-remove — but /settings/statuses (admin-only) lets you change each built-in's ICON app-wide (`setStatusIcon` action; `ensureDefaultStatuses` keeps a chosen icon since it only fills the icon when empty); workspace-scoped (`status.workspaceId`) managed in workspace settings, auto-eligible for new projects in that workspace; project-scoped (`status.projectId`) managed in /projects/[id]/settings, auto-eligible. Built-ins bootstrapped in `src/lib/server/statuses.ts` (`ensureDefaultStatuses`, called from hooks; seeds the 5 default colors) — keep `scripts/seed.ts` in sync. Name uniqueness enforced in code per scope (no DB unique). Each status has an optional `status.color` (hex, tinted pill), an optional `status.icon` (emoji or `iconoir:<name>`, picked via `IconPicker`; built-ins seed default iconoir icons), and an optional `status.description` (≤200 chars). The workspace + project status editors share **`src/lib/components/StatusEditor.svelte`** — a category-grouped editor that **groups statuses by category** (one section per category, gray header bar + a per-category "+" opening an inline create row: color/icon/name/description). Built-in defaults (and, in a project, the inherited workspace statuses) render **read-only** under their category; scope-custom statuses are click-to-edit (name/description/color/icon/category), deletable (disabled while in use), and **draggable to reorder within their category** — drop POSTs `?/reorderStatus` (ordered ids → server rewrites `status.position`, keeping customs after inherited rows). `StatusSelect` (pill + click popover) renders per a **per-view** status display (`view.config.statusDisplay` = `text` (default) | `icon` | `text-icon`, set in the view **Customize** pane, ADR-041) — falling back to the legacy project default (`project.statusDisplay`, no longer editable in the UI) then `text`; threaded as the `statusDisplay` prop through the views + `TaskPanel` (only the active view renders, so the per-view value drives every view). Text shows the name, icon shows `status.icon` via `EntityIcon` (falling back to the name when a status has no icon). In **icon-only** display (and the status has an icon) the pill drops its background + border (`.status-pill.bare`) — just the tinted glyph. Projects have an optional `project.icon` — an emoji OR an `iconoir:<name>` reference (set from the project "…" → Icon submenu's searchable `IconPicker`; ADR: Cross-cutting), rendered via `EntityIcon` in the sidebar, header, and cards. Project structure management lives in the /projects/[id]/settings pane (general/statuses/labels/deps/milestones/grants/delete).
- Views (ADR: Project page): `view` table, one Svelte component per type in `src/lib/components/views/` (table/board/list/dashboard/map/timeline/calendar/**flow**). MULTIPLE views per type allowed; new projects get one Table view (via `createProjectWithDefaults` in `src/lib/server/projects.ts` — always create projects through that helper; it requires a `workspaceId`). The "+" (revealed on viewbar hover) creates a view of a type or unhides a hidden one; type fixed after creation. Right-click a tab → context menu: Rename / Display as (`config.display`: text|text-icon|icon) / Customize | Duplicate / Delete (confirm, real delete). The **Customize** item opens in the shared right-side `SidePane` (ADR: Project page); it edits the ACTIVE view and follows whichever tab is active; choices auto-save (each toggle/blur POSTs `?/updateView`, no Save/Cancel), there's no Type row (fixed at creation), and it has its own real Delete view (no Hide). It also carries a **Status display** select (`config.statusDisplay` = text|icon|text-icon, per-view, ADR-041) for any view type that renders statuses (table/board/list/flow/timeline/calendar). The Customize menu item is a `<button>` with `stopPropagation` (NOT a navigating `<a>`) so the window `closeMenus` click can't race it shut. Project must keep ≥1 visible view. View config is schemaless JSON, persisted per `view.config`; treat keys as optional hints. Map uses Leaflet (client-only import), `task.location` = "lat, lng". TableView cells/headers are `white-space: nowrap` + `.table { width: 100% }`: the table fills the container but keeps column widths and lets the content pane scroll horizontally when the side pane narrows `.proj-page` (no squish) — don't reintroduce wrapping. Table (`.table-view`) and list (`.list-wrap`) roots are capped at `max-width: 1200px` so they don't stretch on wide screens. **Resizable columns**: a `<colgroup>` + per-column drag handles (`.col-resize` on each data column's `<th>` right edge — **incl. `status`** [the `select`/`actions` control columns stay fixed, no handle], `canEditView` only) persist widths to **`config.colWidths`** (`{ colKey: px }`, keys `status|title|priority|assignee|milestone|due|labels|cf:<fieldId>|actions`); once any width exists the table switches to `table-layout: fixed` with an explicit `width` = sum of widths (so columns are authoritative + the pane scrolls) and cells clip; drag seeds all columns from their current rendered widths to avoid a jump, then POSTs `?/updateView` on pointerup. **Group by** (`config.groupBy` = `milestone|status|due|assignee|label`, set via Customize): TableView renders **one `<table>`** with a single shared `<thead>` + one `<tbody>` per group, each led by a spanning **group-header row** (chevron + title + count + hover "+" that creates a task prefilled with the group's value); `groups` derived; the task rows are a `{#snippet groupRows(rows)}`. **One table on purpose** — a single auto-layout makes every column line up across all groups (separate per-group `<table>`s sized their columns independently and misaligned; e.g. Status width differed per group). The "…" column show/hide menu is a single `<thead>` control (`colMenuOpen` boolean; leads with "Create task…", no "Columns" heading). Empty groups are dropped **by default** — gated by `config.hideEmptyGroups` (default true; a **Hide empty groups** chip in the table + list Customize **Group by** section, shown only when a `groupBy` is set, reveals them when off — emptiness honors the view's filters/search/sort since groups derive from the already-filtered rows). The same chip appears in the **board** Customize Group-by section when grouped into swimlanes (`groupBy !== 'status'`), hiding empty **swimlanes** (`visibleLanes` filters `lanes` by tasks in shown statuses); board status **columns** are never hidden (drag-drop targets). `due` uses date buckets (Overdue/Today/Next 7 days/Later/No due date). **`label`** groups by label — a task appears in EACH of its labels' groups (multi-value) + a "No label" group; the group "+" prefills the label (`createTask` accepts an optional `labelId` → inserts `task_label`). **ListView** now supports the SAME **Group by** set (`config.groupBy` = `status|milestone|assignee|due|label`, set via the Customize **list** section) — group-header rows + a `{#snippet rowItem}` per task (no grouping = the flat ordered list). **Sort by** (`config.sortBy` = `priority|order`, set via Customize) orders rows within each group. **BoardView** also supports **Group by** (`config.groupBy` = `status` (default) `|milestone|assignee|label`, set via Customize): columns are ALWAYS statuses; `milestone`/`assignee`/`label` split the board into horizontal **swimlanes** (`lanes` derived — one row per value + a `_none` row), each row a full status-column board (`cellTasks(laneKey, statusId)`). Cross-column drag = `moveTask` (status + position); cross-swimlane drag also `patchTask`es `milestoneId`/`assigneeId` (then `moveTask`) — but **`label` lanes are multi-value, so cross-lane drag changes status/position only** (labels toggled in the pane); each cell's quick-add prefills the status + the lane's field (`labelId` for label lanes). In BOTH views every group/lane header has a leading chevron (`Icon nav-arrow-down/right`) toggling collapse — local component state (`collapsed`/`collapsedLanes`), not persisted. Popovers/menus use the shared `popover` transition (`src/lib/transitions.ts`, fade+rise+scale). TableView **group-header row** puts its chevron in a real `col-select` cell (aligns under the row checkboxes), shows the per-group **task count only when `config.showCount`** (default off, Customize "Show task count"), and the **Title header text is indented** to line up with the task titles. Each group's `<tbody>` renders as its **own bordered + rounded "card"**: the `.table` element has NO perimeter border; instead the frame lives on the edge **cells** — `border-left` on `td:first-child`, `border-right` on `td:last-child`, `border-top`/`border-bottom` + corner radii on the tbody's `tr:first-child`/`tr:last-child` (structural `:global` selectors, so they frame group-row/task-row/sub-row alike; a collapsed group = the group-row is both first+last child → a single fully-rounded bar). Clean **gaps between cards** come from empty spacer pseudo-rows (`.table tbody + tbody::before` = 20px; `thead + tbody::before` = 8px under the header) — they have NO cells, so the per-cell side borders break there (a real `<td>` couldn't do this — the old single-table perimeter border ran full-height and looked like an empty bordered row). **Flow view (ADR-037)**: `FlowView.svelte` (client-only — dynamically imports `FlowCanvas.svelte` in `onMount` so `@xyflow/svelte` never runs at SSR) renders a Svelte Flow dependency graph of top-level tasks (laid out left→right by longest-path dependency depth, NOT milestone-bucketed) + an optional milestone left-lane overlay (`config.flowMilestones`, default on) with dashed membership edges; double-click a milestone node → focused single-milestone task flow with expand/collapse to reveal sub-tasks; node click opens `TaskPanel`. Custom node = `FlowNode.svelte`. Flow takes the standard FilterBar + Customize Sort by / Statuses shown / Show milestones. No layout dep (manual layout), display-only.
- Project page header (ADR: Project page): "…" menu (editors), all items uniform `.menu-item`s = **Create…** (hover fly-out → **Task** opens the New-task pane / **Milestone** opens a focused New-milestone pane; both distinct from the lower **Milestones…** management pane) / **Status** (`project.statusId` — hover fly-out to change; code-validated against defaults + workspace statuses, no FK) / Edit project… / Pin (`project.pinned`, ✦ + sort-first in sidebar) / **Icon** + **Labels** (hover fly-out submenus) / **Milestones…** (opens the shared right-side pane, ADR: Project page) / Delete. Create/Icon/Labels/Status submenus open on HOVER (`.menu-sub:hover > .flyout`), not click-expand. The project title is baseline-aligned with its icon. No Settings/Edit-view buttons, no status filter bar, no add-task bar (board quick-add is the creation affordance). Under the title a **chips row** renders the project's OWN (entity='project') custom fields as two-tone key-value pills `[ name | value ]` (`.kv-pill`/`.kv-key`/`.kv-val`, ADR-041 — replaces the old project-label badges; dependency project links kept) — display-only, resolved by `projectFieldDisplay` (number/date/checkbox/select/person/place/task/files); **rollup** chips show a value computed server-side in the page load (`projectRollupText`, aggregated over the project's tasks — never stored); values edited in project settings; the load splits `listProjectCustomFields` into task-facing + `projectFields` and returns `projectFields`/`projectFieldOptions`/`projectCustomValues`/`projectRollupText`. **Which** project fields show + their **order** is controlled by `project.chipFields` (ordered JSON id array; null = show all with a value, `[]` = none) — set on the custom-fields page's **"Show on project page"** drag-drop bar (`setProjectChipFields` action; REST `PATCH /api/projects/{id}` `chipFields`); empty-valued fields never render.
- Permissions (ADR: Auth/access): visibility = access. Admins see all; others see only workspaces they own / hold a workspace grant on + directly project-granted projects. Guard EVERY project/task read with `canAccessProject` (return 404, not 403 — inaccessible ≡ missing) and filter every list (`accessibleWorkspaceIds` + `grantedProjectIds`). Task editing needs project ACCESS (`canEditTask` ⇒ `canAccessProject`). Structure edits (workspace meta, project CREATE-in-workspace/meta/delete, views, eligible statuses, milestones, project labels) need admin, workspace owner, or a `permission` grant row — check via `canEditWorkspace/Project/View` in EVERY structure mutation (form action + REST). Cascade: workspace owner/grant ⇒ its projects ⇒ their views. View grants do NOT confer visibility. Project grants managed by admins in project settings; workspace grants by admins or the owner in workspace settings. **User rosters are scoped too (don't leak the full user list):** the assignee picker/grouping in a project view loads ONLY `projectAccessUserIds(projectId, workspaceId)` (admins + workspace owner + workspace/project grantees) ∪ already-referenced users (task assignees, person-cf values), NOT every user — a member of one project must not see the whole org's users (incl. emails) or every project name (`allProjects` is access-filtered before return). Settings pages keep the same rule: admins get the full roster (they manage grants → must pick non-members), other editors get the scoped set. There is no "list all users" REST endpoint; grants endpoints are admin-gated and return only existing grantees. **`/projects` (and the sidebar) show only the CURRENT workspace's projects** — scoped by the `workspace` cookie (`cookies.get('workspace')` → first accessible workspace), the same way the sidebar filters; switching workspace re-scopes. Access-filtering alone is not enough — also scope project lists to the current workspace.
- Start / due dates: `task.startDate` (Start-date pill in `TaskPanel`, next to the Due-date pill — both `align="right"` to avoid overflowing the pane), `milestone.startDate` (NewMilestonePane + settings create + Milestones-pane edit, alongside `targetDate`), and `project.startDate` + `project.dueDate` (project settings → General, side by side) — all optional `<input type=date>`s persisted via the same patch/update actions as their existing date fields.
- Milestones (`milestone`, per project, `task.milestoneId` same-project; cols incl. `description`, `startDate`/`targetDate`, `position`) — managed by the shared **`MilestonesManager.svelte`** rendered in BOTH the /projects/[id]/milestones settings page AND the project "…" → Milestones… pane. Each milestone is a **card**: inline-editable name + **description** (textarea, `update({reset:false})`), a **progress bar** (done/total top-level tasks, green at 100%), an **Overdue** badge (targetDate past + <100%), start/target date pickers, a **Depends on** popover (`milestone_dependency`, DFS cycle-checked), trash-delete (confirm), and a **drag handle** to reorder (pointer-based `use:sortable`, touch-capable → `reorderMilestone` action → `reorderMilestones` in `src/lib/server/milestones.ts`; REST `PATCH /api/projects/{id}/milestones` `{order}`). The component posts to `?/updateMilestone`/`?/deleteMilestone`/`?/createMilestone`/`?/setMilestoneDeps`/`?/reorderMilestone` — all present on BOTH routes (settings + project page). Progress is computed by `milestoneProgressByProject` (settings load) or a client map from `data.tasks` (project pane). `canEdit=false` renders read-only. Labels (`label`/`label_group` — workspace-scoped, managed in workspace settings, attached via `project_label`/`task_label` joins; pickers offer only the project's workspace's labels; each label has an optional `color` (hex) + `icon` (emoji/`iconoir:`) set via color swatch + `IconPicker`, rendered as a tinted `LabelChip.svelte` in TaskPanel/Table/List; a label is EITHER workspace-scoped (`label.workspaceId`, shared pool, toggled onto projects via `project_label`) OR **project-scoped** (`label.projectId` — owned by one project, created/edited/deleted in that project's settings Labels section, always available to its tasks; the project view load unions workspace + project-scoped labels), dependencies (`project_dependency`, `task_dependency`, `milestone_dependency` — same project, sub-tasks only on siblings, DFS cycle check). All dependencies are informational only.
- **Essential features (ADR-032)** — features added schema-first (`task.startDate/recurrence/coverFileId`, tables `comment`/`activity`/`notification`/`template`): **search + filter** (`FilterBar.svelte`, pure `taskFilter.ts`/`taskSort.ts`, set lives in `view.config.filters`; views get a pre-filtered `tasks` prop) — facets are **inclusion sets** (ADR-035, supersedes ADR-033): each `filters` array holds the CHECKED (shown) values and a facet filters ONLY when its key is **present** — absent key = inactive (all shown, all options render checked; new views have no `filters` → all checked). `matchTask` keeps a task only if its facet value is in the list (labels: kept if ANY of its labels is checked; `_none` shows unlabeled/unassigned/no-milestone/no-due; empty present array shows nothing). FilterBar/Customize `toggle` seeds the full set on first uncheck + deletes the key when all re-checked; badge = count of unchecked. NOTE: pre-ADR-035 stored filters read backwards until cleared. The FilterBar facet pills live behind an **icon-only `filter-list` button** (right of the search box; lightweight toggle dropdown, NOT the shared `Popover` which would clip the nested facet popovers — badge = active-facet count). Free-text **search also matches custom-field values** (resolved to display labels): the page builds a per-task `cfSearchByTask` map and passes it as `FilterHelpers.searchableText`, folded into `matchTask`'s haystack — **comments + activity** (`server/comments.ts` + `TaskComments.svelte`; `logActivity` from create/patch/setStatus); **notifications** (`server/notifications.ts` + `NotificationBell.svelte`); **timeline** + **calendar** view types (`TimelineView`/`CalendarView.svelte`; also add new types to the server `VIEW_TYPES` allow-list in `server/projects.ts`); **bulk actions** (`selection.svelte.ts` + `BulkActionBar.svelte`; `bulkPatchTasks`/`bulkDeleteTasks`/`bulkReparentToNew`/`bulkSetLabel`; reused for **sub-task bulk edit** in the task pane, incl. a multi-value **Labels** toggle); **templates + recurring** (`server/templates.ts`, `recurrence.ts`); **CSV export** (`/api/projects/{id}/export`); **photo capture** (`TaskAttachments.svelte`, `/api/tasks/{id}/files`); **mobile** (`mobile.css`). New REST endpoints are documented in `static/llms.txt`. `Popover.svelte` has an `up` prop for bottom-anchored bars.

## Conventions

- **Icons (ADR: Cross-cutting)**: use **iconoir** via `Icon.svelte` (`<Icon name="nav-arrow-down" size={14} />`) for UI chrome — names come from the external sprite `static/iconoir.svg` (regenerate with `npm run icons:build` after bumping the `iconoir` dep; the searchable list is `src/lib/iconoirNames.ts`). Don't hand-write unicode glyph chars for affordances (…, ▸, ×, chevrons, view-type icons). Stored entity icon values render via `EntityIcon` (emoji or `iconoir:<name>`). Keep semantic custom glyphs (PriorityIcon bars, BoardView status dots).
- **Color picker (ADR-040)**: pick colors with **`src/lib/components/ColorPicker.svelte`** — the color twin of `IconPicker` (280px panel-in-`Popover`, controlled `{ value, onSelect(hex), onRemove? }`, caller owns persistence). Shows a 21-swatch `COLOR_PRESETS` grid (Tailwind-500/600; the 5 status defaults are members) + a custom-hex input + a styled native `<input type=color>` "Custom" hatch. `normalizeHex` emits lowercase **6-digit** `#rrggbb` (server `parseColor` is 6-digit-only). a11y: `role=radiogroup` of `role=radio` swatches, roving tabindex, `aria-checked` (NOT `aria-pressed`). Used at the 4 color sites (StatusEditor, CustomFieldEditor, project labels, workspace labels) via a `Popover` with a `.cp-swatch` trigger — form sites add a hidden `<input name="color">`; label-edit sites call the `patchLabel`/`patchProjectLabel` fetch helper from `onSelect`. Don't reintroduce raw `<input type="color">`/`.color-in`.
- **Tooltips (ADR-034)**: `use:tooltip={text | { text, placement, delay }}` from `src/lib/tooltip.ts` — a Svelte action backed by ONE shared floating node (themed via `.app-tooltip` in `app.css`, inverse bubble, ≤120ms fade, reduced-motion aware). Shows on hover (mouse only) + keyboard focus after a delay, flips at the viewport edge, hides on leave/click/Escape/scroll, and **strips the host's native `title`** (no OS double). Use it for icon-only affordances (buttons, drag/resize handles, avatars, badges, relative timestamps) instead of native `title=`. (`IconPicker`'s large emoji/icon grid keeps native `title` for perf.)
- **No decorative glyphs** (ADR: Project page): don't prefix entity names with ornaments (`◇` milestone, `●` label dot, `⛓` project) — show the plain name. (A `⛓ N` blocker *count* badge is fine; it's a count, not a name prefix.) Statuses keep their color (the pill tint / swatch carries data, not decoration).
- **i18n**: every user-facing UI string goes through `$t('English text')` from `$lib/i18n` — English strings are the keys (en = passthrough). App is English-only (Vietnamese removed); the `$t` wrapper + `registerDictionary` are kept so a locale can be re-added by registering a dictionary. Never translate DB content (status/label/project/task/user names).
- **llms.txt**: `static/llms.txt` documents the REST API for agents (served at `/llms.txt`). KEEP IT UPDATED whenever a REST endpoint, request/response field, auth, or status/workspace rule changes.

- **ADR.md**: every architecture/design decision (new dependency, schema approach, auth model, API shape, theming strategy, etc.) gets a new `ADR-NNN` record — context, decision, consequences. Update an existing record's status (`Superseded by ADR-NNN`) instead of rewriting history.

- **Theming (ADR: Cross-cutting)**: Tailwind 4 + DaisyUI 5. `src/app.css` = `@import "tailwindcss"; @plugin "daisyui" { themes: light --default, dark --prefersdark; }` + fonts (Inter/IBM Plex Mono) + a **token bridge** (legacy `--color-*`/`--sp-*`/`--dur-*`/`--radius` vars mapped onto DaisyUI theme tokens like `--color-base-100`/`--color-base-content`/`--color-base-300`, so component-scoped `<style>` blocks follow the active theme automatically). Prefer DaisyUI component classes (`btn btn-primary`, `input`, `select`, `card`, `badge badge-neutral`, `alert alert-error`, `menu`, `modal`, `dropdown`) and Tailwind utilities; don't hardcode colors — use DaisyUI tokens or the bridged `var(--color-*)`. `.card`, `.label`, `.field`, `.u-*` are thin app-specific helpers kept in app.css. Focus rings are removed app-wide (`:focus, :focus-visible { outline: none }` in app.css). Theme = `theme` cookie, applied to `<html data-theme>` at SSR via `hooks.server.ts` `transformPageChunk`; toggled from the topbar **Appearance** dropdown in `(app)/+layout.svelte` (a `half-moon` button → `role=menu` with **Dark mode** + **High contrast** `menuitemcheckbox` rows; `theme`/`contrast` seeded from root `+layout.server.ts` load). Add a theme = add it to the `themes:` list (and the `THEMES` arrays in hooks + root layout load). **High contrast (ADR-038)** is a SEPARATE axis: a `contrast` cookie → `<html data-contrast="high">` at SSR (same `transformPageChunk`), toggled from the same topbar **Appearance** menu AND the checkbox in /settings → **Theming**; `[data-contrast='high']` in app.css overrides the bridge + DaisyUI tokens to black-on-white (incl. forcing DaisyUI's faint `.table` cell borders solid). Data-driven accents (status/label colors) keep their inline colors.
- Tabs for indentation; single quotes.
- Single instance, access-scoped (ADR: Auth/access): users see only workspaces/projects they can access (owner, grants, admin). Workspace ownership/grants are the sharing primitive.
- Destructive confirms: never `onsubmit` + `preventDefault()` on a `use:enhance` form (enhance's fetch fires anyway) — use the enhance callback's `cancel()`, OR the centered confirm modal. **Confirm modal**: `confirmDialog(message, { confirmLabel?, danger? })` from `src/lib/confirm.svelte.ts` returns a `Promise<boolean>`; `src/lib/components/ConfirmModal.svelte` (mounted once in the `(app)` shell, like Toaster) renders a page-centered dialog (fixed backdrop, Escape/backdrop = cancel, Enter = confirm). Pattern for an enhance form: make the trigger a `type="button"`, `await confirmDialog(...)`, then `form?.requestSubmit()` (don't await inside enhance — its fetch already fired). Task + sub-task deletes (`TaskPanel`) use this (danger, "Delete this task[ and its sub-tasks]?" / "Delete this sub-task?").
- Motion: no animation library (ADR: Cross-cutting). Use `svelte/transition`/`animate:flip` with the `--dur-fast/--dur/--dur-slow` tokens; everything ≤200ms.
- **Touch / mobile (ADR-042)**: drag-and-drop + right-click must work on touch — native HTML5 DnD (`draggable`/`ondragstart`/`ondrop`) and `oncontextmenu` are mouse-only. Use the in-house **pointer-event** primitives instead: **`use:sortable`** (`src/lib/sortable.ts`) for list reorder — container action, items get `data-sortable-id`, drag from an optional `[data-sortable-handle]`; mouse drags on a move threshold, touch engages after a still HOLD (so a swipe still scrolls), calls `onReorder(ids)`; and **`use:longpress`** (`src/lib/longpress.ts`) which fires the SAME handler as the mouse `oncontextmenu` (keep both on the element). Both swallow the trailing click. **BoardView** card drag is bespoke pointer-based (hold-to-engage; `document.elementFromPoint` hit-tests `data-lane`/`data-status`/`data-index`; the dragged card gets `pointer-events:none`). Add `-webkit-touch-callout:none` to long-pressable/draggable elements. Adopted: view-tab menu (mouse still drag-reorders tabs; touch tab-reorder is out of scope), BoardView col-head menu + card drag, StatusEditor (one sortable per category), CustomFieldEditor, chip-fields Show bar (tap to add/remove, handle-drag to reorder). Don't reintroduce native `draggable` for new reorder UI; resize handles already use pointer events.
- `.env` is gitignored; secrets read via `$env/dynamic/private` (server) or `process.env` (scripts).
