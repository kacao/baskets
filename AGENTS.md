# CLAUDE.md

Baskets — single-tenant project management app (projects → tasks → sub-tasks). Minimalist. UI built on Tailwind 4 + DaisyUI 5 (ADR: Cross-cutting). Product scope: `PRD.md`. Architecture decisions: `ADR.md`.

## Commands

```bash
npm run dev          # dev server on :5173
npm run build        # production build (adapter-node → build/)
npm run check        # svelte-kit sync + svelte-check (run before committing)
npm run db:push      # apply src/lib/server/db/schema.ts to SQLite (drizzle-kit push)
npm run db:seed      # seed admin/demo users + sample data (idempotent; skips if admin exists)
npm run start        # serve production build via custom server.js (adapter-node handler + /ws WebSocket; reads .env)
```

No test suite yet. Verify changes with `npm run check` + manual smoke test.

db:push gotcha: when one push both rebuilds a table (e.g. dropping a unique) AND adds a column to it, drizzle-kit generates a broken `INSERT … SELECT new_column FROM old_table` and dies mid-push, leaving the DB partially migrated. Back up the .db first; if it happens, apply the missing `ALTER TABLE … ADD COLUMN` by hand, then re-run `npx drizzle-kit push --force` until "Changes applied".

## Stack

- SvelteKit 2 + **Svelte 5 runes** (`$props`, `$state`, `$derived` — no legacy `export let` / `$:`)
- UI: Tailwind 4 (`@tailwindcss/vite`) + DaisyUI 5 (see Theming under Conventions)
- SQLite via better-sqlite3 + Drizzle ORM (`DATABASE_URL` in `.env`, default `./data/baskets.db` — DB lives in the gitignored `data/` dir, which is kept in git via `data/.gitkeep`)
- BetterAuth 1.6 with `twoFactor` (TOTP) and `admin` plugins

## Architecture

- `src/lib/server/db/schema.ts` — ALL tables: BetterAuth tables (user, session, account, verification, twoFactor) AND app tables (workspace, project, task, …). BetterAuth field requirements change between versions; if it errors about a missing field, add the column here and `npm run db:push`. Keep `scripts/seed.ts`'s auth config in sync with `src/lib/server/auth.ts`.
- Workspaces: every project belongs to one (`project.workspaceId`); workspaces own custom statuses and ALL labels/groups. `workspace.ownerId` + `permission` grants (`resourceType 'workspace'`) gate workspace structure edits and cascade to its projects/views — and per ADR: Auth/access they also define VISIBILITY (non-admins only see workspaces they own/are granted, plus directly granted projects). `ensureDefaultWorkspace` (hooks) bootstraps a "Default" workspace and adopts pre-workspace rows; the nullable `workspaceId` FKs exist only for db:push on populated tables — code treats them as required. Workspace CRUD at /workspaces; per-workspace settings pane (general/statuses/labels/grants/delete) at /workspaces/[id]/settings. A workspace is deletable only when empty; the last one can't be deleted. A user's FIRST workspace (they own none yet — `isFirstWorkspaceForUser`) is seeded with sample projects/milestones/tasks via `seedWorkspaceSamples` (in `projects.ts`). The sidebar header is a workspace switcher (current workspace name + chevron; `workspace` cookie, fallback = first workspace) and the sidebar project list is scoped to the current workspace. Sidebar workspace name / Projects link / project list share one left edge.
- `src/lib/server/auth.ts` — BetterAuth instance. `src/lib/auth-client.ts` — browser client (used for auth + admin actions). `src/hooks.server.ts` — populates `locals.user` / `locals.session` on every request.
- Routes: `(auth)` group = login/register/two-factor (redirects away if signed in); `(app)` group = everything else (redirects to /login if not; its `+layout.svelte` is the shell with sidebar). `/admin` additionally requires `locals.user.role === 'admin'` (enforced in its `+page.server.ts`).
- Data mutations: SvelteKit form actions in `+page.server.ts` files (projects, tasks). Exception: auth/2FA/admin-user actions go through `authClient` (BetterAuth API) client-side.
- REST API: `src/routes/api/{projects,tasks}` `+server.ts` endpoints (GET/POST/PATCH/DELETE). Auth via session cookie OR `Authorization: Bearer bsk_…` API key. Keys are managed per-user in /settings (SHA-256 hash stored in `api_key` table, plaintext shown once); `src/lib/server/api-keys.ts` generates/resolves them, `src/hooks.server.ts` resolves the bearer header into `locals.user`. Shared validation helpers in `src/lib/server/api.ts`.
- Integrations: `integration` table (one row per `type`, single tenant), managed at /integrations. `src/lib/server/integrations/index.ts` exports `dispatchEvent(event)` — call it fire-and-forget (`void dispatchEvent(...)`, never await) after mutations; it never throws. Events: `project.created`, `task.created`, `task.completed`. Slack uses an incoming webhook (URL must start with `https://hooks.slack.com/`; never sent back to the client). New integration = new module in `src/lib/server/integrations/` + a case in `dispatchEvent`.
- Realtime (ADR: Realtime): WebSockets at `/ws` for live updates + presence. Server transport is **plain ESM** `src/lib/server/realtime/attach.js` (no `$lib`/TS), shared by the Vite dev/preview plugin (`vite.config.ts`) and prod `server.js` (`npm start` runs it, not `build/index.js`). Auth on upgrade + per-project `subscribe` are checked by forwarding cookies to `GET /api/me` and `GET /api/projects/{id}` (404 = no access, ADR: Auth/access) — the WS layer imports no app modules. Live connections live on `globalThis.__basketsRealtime` (shared across dev module realms); app mutations call `broadcastProjectChange(projectId, actorId)` (from `src/lib/server/realtime/hub.ts`, fire-and-forget, never throws) after EVERY project form action + task/project REST write. Sync = invalidate-and-refetch: clients debounce `invalidateAll()` on a `changed` ping (reuses all `load()` permission filters; no deltas). Browser client + presence store: `src/lib/realtime.svelte.ts` (reconnect + 30s heartbeat); presence avatars render in the project header.
- Side pane (ADR: Project page): `src/lib/components/SidePane.svelte` is the ONE reusable right-pane chrome (slide-in, Escape-close, scrolling body, `title` or `header` snippet + × ). Task editing (`TaskPanel`), the view **Customize** pane, and the **Milestones** pane all render inside it. Task edit opens the pane on title click in ALL views (Table/List/Board) — no inline editing. **Sub-tasks open their own pane too**: clicking a sub-task title (a table sub-row, or a row in the pane's sub-task table) opens it via the view's `onSelectTask` callback (views' `selected` no longer filters out `parentId`); the sub-task's pane shows a `← <parent>` breadcrumb (`onSelectTask` back to the parent) and hides the Sub-tasks section (depth-1, no children). It is an **in-flow flex sibling of `.content`** (`position: relative; flex: 0 0 var(--pane-w)`), portaled (`use:portal`) up into the app shell's `[data-pane-host]` (the `.main-body` flex row in `(app)/+layout.svelte`). The shell is fixed-height (`.shell { height: 100dvh }`, `.main`/`.main-body` `min-height:0`) and `.content` is `overflow: scroll`, so content owns its scrollbars and the pane sits beside it. Opening the pane resizes `.content` but **not its children**: SidePane mirrors its live width to `--pane-w` on the portal host (`.main-body`), and `.content > *` gets `min-width: calc(100% + var(--pane-w,0px))`, so children keep their pre-open width and `.content` scrolls horizontally instead of reflowing (the 900px overlay breakpoint resets this to `100%`). Pane top aligns to the topbar bottom (≈49px), left border, no shadow. **Resizable** via a left-edge drag handle (sets `--pane-w` on the aside too, clamps 320…min(900,92vw), persists in `localStorage('sidePaneWidth')`). Under 900px = fixed full-width overlay, handle hidden. (Tradeoff: a popover opening past the content's right edge can clip at the pane boundary.) Only ONE pane is open at a time — `src/lib/sidePaneRegistry.ts` (`openPane`/`closePane`, called from SidePane's onMount/onDestroy) closes the previously-open pane when another opens, across all owners (task pane + Customize + Milestones).
- Tasks: one level of nesting only — `task.parentId` set ⇒ sub-task; sub-tasks cannot have children (enforced in createTask AND patchTask's re-parent branch). `patchTask` accepts **`parentId`** to nest an existing task under another (childless + same-project + parent-is-top-level checks), or `null`/empty to **detach a sub-task back to top-level**; REST `PATCH /api/tasks/{id}` accepts the same `parentId` (same guards). Setting a parent to a `done`-category status cascades to its subs (cascade fires on the parent's status-change only — re-parenting under an already-done parent does NOT force the moved task done, matching `createTask`). Task pane (`TaskPanel` in `SidePane`, ADR: Project page): full-width Title + Description (auto-save on blur), a **pills row** between them — Priority/Assignee/Milestone/Order/Location/Blocked-by/Due-date (+ a **Parent** pill shown on sub-tasks only — a `patchTask` `parentId` Popover to move under another top-level task or detach to top-level) — each a `Popover.svelte` (pill + click panel; Milestone/Location panels have search + inline create; the Due-date pill's panel is a `<input type=date>` + Clear). Pills + title/desc persist via the **`patchTask`** action (patches only fields present in the form via `form.has`, so one field never clobbers another). The Description form uses `enhance … update({ reset: false })` — a `<textarea>`'s value comes from child text, so the default `form.reset()` would wipe it to an empty defaultValue on a client-rendered pane. `createMilestone`/`createLocation` accept an optional `taskId` to create-and-assign atomically. **Sub-tasks render as stacked blocks** (`.sub-list`, only when the open task is a parent — `!task.parentId`): each sub-task = a head row (Status pill + clickable title that opens it + delete ×) with its editable property pills (Priority/Assignee/Milestone/Due, each a `patchTask` Popover on that sub-task) **stacked underneath** the title (indented past the status). In the main TableView, expanded sub-rows now render the **same columns as parent rows** (status/title-with-chevron-spacer/optional cols/cf cols) so a sub-task's custom-field values are visible + column-aligned in the project view (a cf cell shows the value only when the field `appliesTo` sub-tasks). The **"+ Add sub-task"** affordance is a `Popover` search (same pattern as Milestone/Location): it lists childless top-level tasks (re-parent via `patchTask` `parentId`) and, on no match, a **Create task "&lt;term&gt;"** button (new sub-task via `createTask` `parentId`). The pane footer (`.pane-footer`) has **Delete task** (confirm modal) + **Close** (`onClose`).
- Locations (ADR: Domain model): project-scoped `location` table (`title` required; `address`, `latitude`, `longitude` all optional/nullable — a location can be a plain named place with no coords; map plots only those with both lat+lng). The task Location/Milestone popovers show a **Create "&lt;term&gt;"** button only when the search has text and no match (no verbose inline form). `task.locationId` FK (`set null` on delete). Managed (CRUD) in project settings; picked/created from the task Location pill. Map views plot from the picked location's coords, falling back to the legacy freeform `task.location` text (kept for old data, no longer editable in the pane). Create/edit/delete need `canEditProject`; *picking* needs only `canEditTask`.
- Custom fields (ADR: Domain model): project-scoped typed fields on **tasks OR the project itself**. `custom_field.entity` = `task` (default; values in `task_custom_value`) | `project` (values in `project_custom_value`, composite PK `(projectId,fieldId)`). The /projects/[id]/settings Custom fields editor has **[Task | Project] tabs** (`CustomFieldEditor` filters its list + sets `entity` on create); project-entity fields' VALUES are edited in a "Project fields" section there via `CustomFieldValue` (`formAction="?/patchProjectCustomValues"` → `writeProjectCustomValues`, the project-side mirror of `writeTaskCustomValues`). Project-entity fields are filtered OUT of the task-facing project-view load + rejected by `writeTaskCustomValues` (entity guard) so they never appear on tasks. Tables: `custom_field` (id/projectId/`entity`/name/`type`/`config` schemaless JSON/`appliesTo` (task entity only)/position — `type` ∈ text|number|select|date|person|files|task|checkbox|email|phone|place|url|**rollup** (rollup = a computed, display-only aggregate: `config.{relation,targetFieldId,formula}` — relation ∈ blocked-by|blocking|sub-task|task (project-entity fields force `task` = all project tasks), formula ∈ count|sum|average|min|max; computed client-side via `computeTaskRollup`/`rollupAggregate`, never stored — `writeTaskCustomValues` ignores it; threaded as a `rollupText` prop into `CustomFieldValue` from TaskPanel/TableView/project-settings); name unique-per-project in code; **type immutable after create**; **`appliesTo`** = `all` (default) | `tasks` (top-level only) | `subtasks`, set in the editor, enforced by `fieldAppliesTo(field, isSubtask)` — filters which fields show in TaskPanel (per the open task's level), NewTaskPane (top-level), and Table columns (`cfCols` = top-level-applicable; sub-rows show a field's cell only when it also applies to sub-tasks; `subtasks`-only fields have no table column and are pane-only); `writeTaskCustomValues` rejects a value whose field doesn't apply to the task's level), `custom_field_option` (select members, enum-with-color/icon like `label`), `task_custom_value` (composite PK `(taskId,fieldId)`, single `value` TEXT — scalar OR a JSON id-array for the multi-capable types select/person/place/files/task (a `task` field with `config.multi` links a LIST of same-project tasks, e.g. an "Inspection" field → many inspection tasks; pre-multi scalar `task` ids are recovered as 1-element arrays by `decodeValue` and the server's lenient parse); absent row = no value), `file` (uploads). Client-safe helpers in `src/lib/customFields.ts` (`CUSTOM_FIELD_TYPES`, `defaultConfig`/`sanitizeConfig`, `decodeValue`, `formatNumber`/`formatDate`), server `src/lib/server/customFields.ts` re-exports + `writeTaskCustomValues` (the ONE validated write path used by `patchTask`/`createTask` form actions AND REST — validates every ref against `field.projectId`) + `customValuesByTask`/`apiCustomFieldEntries` (REST). Field + option CRUD (drag-reorder) lives in /projects/[id]/settings via shared **`CustomFieldEditor.svelte`**; values render+edit through one **`CustomFieldValue.svelte`** with `mode` `pill` (TaskPanel, posts `cf_<fieldId>` to `?/patchTask`), `cell` (optional Table columns keyed `cf:<fieldId>`, toggled like the built-in column show/hide), `input` (NewTaskPane, hidden `cf_<fieldId>` read by `createTask`). v1 is **display-only** (no sort/group/filter by custom fields). Number `custom` format string is stored but NOT interpreted (rendered via `Intl.NumberFormat`; `custom` shows raw). Deferred **formula**/**button** slot in via type+config (no schema change). Threaded through the project page load → views → TaskPanel/NewTaskPane as `customFields`/`customFieldOptions`/`taskCustomValues`/`files`.
- Files (ADR: Domain model): `files`-type custom-field uploads on **local disk** under `UPLOADS_DIR` (default `./data/uploads/`, created on demand, gitignored — see `src/lib/server/uploads.ts`). Served ONLY via access-gated `GET /api/files/[id]` (streams; `canAccessProject` → 404; `nosniff` + CSP `sandbox`, non-raster-image as `attachment` — no stored-XSS from our origin). `POST /api/files` (multipart: `fieldId` + `file` [+ `taskId`, validated same-project]; 10 MB cap; denylist of exec + active-markup exts (svg/html/xml/js/…); **MIME server-derived from extension** via `mimeForExt`, NOT the client claim; project derived from the field), `DELETE /api/files/[id]` (unlinks row + disk + strips id from referencing values). `storagePath` is never sent to the client. Deleting a files-field unlinks its bytes first (`deleteFilesForField`).
- Statuses: table-driven (`status` + `project_status` eligibility). Behavior keys off `status.category` (`backlog|planned|in-progress|completed|canceled` — one per default status; `completed` = the "done" bucket that completes sub-tasks; `backlog` = new-task default), never the name. Categories + display labels are in the client-safe `src/lib/statuses.ts` (`STATUS_CATEGORIES`/`categoryLabel`/`DONE_CATEGORY`/`INITIAL_CATEGORY`), re-exported by `src/lib/server/statuses.ts`; `ensureDefaultStatuses` reconciles built-in name/category/color on boot. Three scopes: app-wide = exactly the five built-ins (Backlog/Planned/In progress/Completed/Canceled), FIXED name/category/no-add-remove — but /settings/statuses (admin-only) lets you change each built-in's ICON app-wide (`setStatusIcon` action; `ensureDefaultStatuses` keeps a chosen icon since it only fills the icon when empty); workspace-scoped (`status.workspaceId`) managed in workspace settings, auto-eligible for new projects in that workspace; project-scoped (`status.projectId`) managed in /projects/[id]/settings, auto-eligible. Built-ins bootstrapped in `src/lib/server/statuses.ts` (`ensureDefaultStatuses`, called from hooks; seeds the 5 default colors) — keep `scripts/seed.ts` in sync. Name uniqueness enforced in code per scope (no DB unique). Each status has an optional `status.color` (hex, tinted pill), an optional `status.icon` (emoji or `iconoir:<name>`, picked via `IconPicker`; built-ins seed default iconoir icons), and an optional `status.description` (≤200 chars). The workspace + project status editors share **`src/lib/components/StatusEditor.svelte`** — a category-grouped editor that **groups statuses by category** (one section per category, gray header bar + a per-category "+" opening an inline create row: color/icon/name/description). Built-in defaults (and, in a project, the inherited workspace statuses) render **read-only** under their category; scope-custom statuses are click-to-edit (name/description/color/icon/category), deletable (disabled while in use), and **draggable to reorder within their category** — drop POSTs `?/reorderStatus` (ordered ids → server rewrites `status.position`, keeping customs after inherited rows). `StatusSelect` (pill + click popover) renders per **`project.statusDisplay`** (`text` (default) | `icon` | `text-icon`, set in project settings → Status display) — text shows the name, icon shows `status.icon` via `EntityIcon` (falling back to the name when a status has no icon), threaded as a `statusDisplay` prop through the views + `TaskPanel`. In **icon-only** display (and the status has an icon) the pill drops its background + border (`.status-pill.bare`) — just the tinted glyph. Projects have an optional `project.icon` — an emoji OR an `iconoir:<name>` reference (set from the project "…" → Icon submenu's searchable `IconPicker`; ADR: Cross-cutting), rendered via `EntityIcon` in the sidebar, header, and cards. Project structure management lives in the /projects/[id]/settings pane (general/statuses/labels/deps/milestones/grants/delete).
- Views (ADR: Project page): `view` table, one Svelte component per type in `src/lib/components/views/` (table/board/list/dashboard/map). MULTIPLE views per type allowed; new projects get one Table view (via `createProjectWithDefaults` in `src/lib/server/projects.ts` — always create projects through that helper; it requires a `workspaceId`). The "+" (revealed on viewbar hover) creates a view of a type or unhides a hidden one; type fixed after creation. Right-click a tab → context menu: Rename / Display as (`config.display`: text|text-icon|icon) / Customize | Duplicate / Delete (confirm, real delete). The **Customize** item opens in the shared right-side `SidePane` (ADR: Project page); it edits the ACTIVE view and follows whichever tab is active; choices auto-save (each toggle/blur POSTs `?/updateView`, no Save/Cancel), there's no Type row (fixed at creation), and it has its own real Delete view (no Hide). The Customize menu item is a `<button>` with `stopPropagation` (NOT a navigating `<a>`) so the window `closeMenus` click can't race it shut. Project must keep ≥1 visible view. View config is schemaless JSON, persisted per `view.config`; treat keys as optional hints. Map uses Leaflet (client-only import), `task.location` = "lat, lng". TableView cells/headers are `white-space: nowrap` + `.table { width: 100% }`: the table fills the container but keeps column widths and lets the content pane scroll horizontally when the side pane narrows `.proj-page` (no squish) — don't reintroduce wrapping. Table (`.table-view`) and list (`.list-wrap`) roots are capped at `max-width: 1200px` so they don't stretch on wide screens. **Resizable columns**: a `<colgroup>` + per-column drag handles (`.col-resize` on each resizable `<th>`'s right edge, `canEditView` only) persist widths to **`config.colWidths`** (`{ colKey: px }`, keys `status|title|priority|assignee|milestone|due|labels|cf:<fieldId>|actions`); once any width exists the table switches to `table-layout: fixed` with an explicit `width` = sum of widths (so columns are authoritative + the pane scrolls) and cells clip; drag seeds all columns from their current rendered widths to avoid a jump, then POSTs `?/updateView` on pointerup. **Group by** (`config.groupBy` = `milestone|status|due|assignee|label`, set via Customize): TableView renders **one `<table>`** with a single shared `<thead>` + one `<tbody>` per group, each led by a spanning **group-header row** (chevron + title + count + hover "+" that creates a task prefilled with the group's value); `groups` derived; the task rows are a `{#snippet groupRows(rows)}`. **One table on purpose** — a single auto-layout makes every column line up across all groups (separate per-group `<table>`s sized their columns independently and misaligned; e.g. Status width differed per group). The "…" column show/hide menu is a single `<thead>` control (`colMenuOpen` boolean; leads with "Create task…", no "Columns" heading). Empty groups are dropped, `due` uses date buckets (Overdue/Today/Next 7 days/Later/No due date). **`label`** groups by label — a task appears in EACH of its labels' groups (multi-value) + a "No label" group; the group "+" prefills the label (`createTask` accepts an optional `labelId` → inserts `task_label`). **ListView** now supports the SAME **Group by** set (`config.groupBy` = `status|milestone|assignee|due|label`, set via the Customize **list** section) — group-header rows + a `{#snippet rowItem}` per task (no grouping = the flat ordered list). **Sort by** (`config.sortBy` = `priority|order`, set via Customize) orders rows within each group. **BoardView** also supports **Group by** (`config.groupBy` = `status` (default) `|milestone|assignee|label`, set via Customize): columns are ALWAYS statuses; `milestone`/`assignee`/`label` split the board into horizontal **swimlanes** (`lanes` derived — one row per value + a `_none` row), each row a full status-column board (`cellTasks(laneKey, statusId)`). Cross-column drag = `moveTask` (status + position); cross-swimlane drag also `patchTask`es `milestoneId`/`assigneeId` (then `moveTask`) — but **`label` lanes are multi-value, so cross-lane drag changes status/position only** (labels toggled in the pane); each cell's quick-add prefills the status + the lane's field (`labelId` for label lanes). In BOTH views every group/lane header has a leading chevron (`Icon nav-arrow-down/right`) toggling collapse — local component state (`collapsed`/`collapsedLanes`), not persisted. Popovers/menus use the shared `popover` transition (`src/lib/transitions.ts`, fade+rise+scale).
- Project page header (ADR: Project page): "…" menu (editors), all items uniform `.menu-item`s = **Create…** (hover fly-out → **Task** opens the New-task pane / **Milestone** opens a focused New-milestone pane; both distinct from the lower **Milestones…** management pane) / **Status** (`project.statusId` — hover fly-out to change; code-validated against defaults + workspace statuses, no FK) / Edit project… / Pin (`project.pinned`, ✦ + sort-first in sidebar) / **Icon** + **Labels** (hover fly-out submenus) / **Milestones…** (opens the shared right-side pane, ADR: Project page) / Delete. Create/Icon/Labels/Status submenus open on HOVER (`.menu-sub:hover > .flyout`), not click-expand. The project title is baseline-aligned with its icon. No Settings/Edit-view buttons, no status filter bar, no add-task bar (board quick-add is the creation affordance).
- Permissions (ADR: Auth/access): visibility = access. Admins see all; others see only workspaces they own / hold a workspace grant on + directly project-granted projects. Guard EVERY project/task read with `canAccessProject` (return 404, not 403 — inaccessible ≡ missing) and filter every list (`accessibleWorkspaceIds` + `grantedProjectIds`). Task editing needs project ACCESS (`canEditTask` ⇒ `canAccessProject`). Structure edits (workspace meta, project CREATE-in-workspace/meta/delete, views, eligible statuses, milestones, project labels) need admin, workspace owner, or a `permission` grant row — check via `canEditWorkspace/Project/View` in EVERY structure mutation (form action + REST). Cascade: workspace owner/grant ⇒ its projects ⇒ their views. View grants do NOT confer visibility. Project grants managed by admins in project settings; workspace grants by admins or the owner in workspace settings.
- Start / due dates: `task.startDate` (Start-date pill in `TaskPanel`, next to the Due-date pill — both `align="right"` to avoid overflowing the pane), `milestone.startDate` (NewMilestonePane + settings create + Milestones-pane edit, alongside `targetDate`), and `project.startDate` + `project.dueDate` (project settings → General, side by side) — all optional `<input type=date>`s persisted via the same patch/update actions as their existing date fields.
- Milestones (`milestone`, per project, `task.milestoneId` same-project) — managed both in /projects/[id]/settings AND in the project "…" → Milestones… pane; milestones may depend on other milestones of the same project (`milestone_dependency`, DFS cycle-checked, edited in the Milestones pane). Labels (`label`/`label_group` — workspace-scoped, managed in workspace settings, attached via `project_label`/`task_label` joins; pickers offer only the project's workspace's labels; each label has an optional `color` (hex) + `icon` (emoji/`iconoir:`) set via color swatch + `IconPicker`, rendered as a tinted `LabelChip.svelte` in TaskPanel/Table/List; a label is EITHER workspace-scoped (`label.workspaceId`, shared pool, toggled onto projects via `project_label`) OR **project-scoped** (`label.projectId` — owned by one project, created/edited/deleted in that project's settings Labels section, always available to its tasks; the project view load unions workspace + project-scoped labels), dependencies (`project_dependency`, `task_dependency`, `milestone_dependency` — same project, sub-tasks only on siblings, DFS cycle check). All dependencies are informational only.
- **Essential features (ADR-032)** — features added schema-first (`task.startDate/recurrence/coverFileId`, tables `comment`/`activity`/`notification`/`saved_filter`/`template`): **search + filter** (`FilterBar.svelte`, pure `taskFilter.ts`/`taskSort.ts`, set lives in `view.config.filters`; views get a pre-filtered `tasks` prop) + **saved filters**; **comments + activity** (`server/comments.ts` + `TaskComments.svelte`; `logActivity` from create/patch/setStatus); **notifications** (`server/notifications.ts` + `NotificationBell.svelte`); **timeline** + **calendar** view types (`TimelineView`/`CalendarView.svelte`; also add new types to the server `VIEW_TYPES` allow-list in `server/projects.ts`); **bulk actions** (`selection.svelte.ts` + `BulkActionBar.svelte`; `bulkPatchTasks`/`bulkDeleteTasks`/`bulkReparentToNew`; reused for **sub-task bulk edit** in the task pane); **templates + recurring** (`server/templates.ts`, `recurrence.ts`); **CSV export** (`/api/projects/{id}/export`); **photo capture** (`TaskAttachments.svelte`, `/api/tasks/{id}/files`); **mobile** (`mobile.css`). New REST endpoints are documented in `static/llms.txt`. `Popover.svelte` has an `up` prop for bottom-anchored bars.

## Conventions

- **Icons (ADR: Cross-cutting)**: use **iconoir** via `Icon.svelte` (`<Icon name="nav-arrow-down" size={14} />`) for UI chrome — names come from the external sprite `static/iconoir.svg` (regenerate with `npm run icons:build` after bumping the `iconoir` dep; the searchable list is `src/lib/iconoirNames.ts`). Don't hand-write unicode glyph chars for affordances (…, ▸, ×, chevrons, view-type icons). Stored entity icon values render via `EntityIcon` (emoji or `iconoir:<name>`). Keep semantic custom glyphs (PriorityIcon bars, BoardView status dots).
- **No decorative glyphs** (ADR: Project page): don't prefix entity names with ornaments (`◇` milestone, `●` label dot, `⛓` project) — show the plain name. (A `⛓ N` blocker *count* badge is fine; it's a count, not a name prefix.) Statuses keep their color (the pill tint / swatch carries data, not decoration).
- **i18n**: every user-facing UI string goes through `$t('English text')` from `$lib/i18n` — English strings are the keys (en = passthrough). App is English-only (Vietnamese removed); the `$t` wrapper + `registerDictionary` are kept so a locale can be re-added by registering a dictionary. Never translate DB content (status/label/project/task/user names).
- **llms.txt**: `static/llms.txt` documents the REST API for agents (served at `/llms.txt`). KEEP IT UPDATED whenever a REST endpoint, request/response field, auth, or status/workspace rule changes.

- **ADR.md**: every architecture/design decision (new dependency, schema approach, auth model, API shape, theming strategy, etc.) gets a new `ADR-NNN` record — context, decision, consequences. Update an existing record's status (`Superseded by ADR-NNN`) instead of rewriting history.

- **Theming (ADR: Cross-cutting)**: Tailwind 4 + DaisyUI 5. `src/app.css` = `@import "tailwindcss"; @plugin "daisyui" { themes: light --default, dark --prefersdark; }` + fonts (Inter/IBM Plex Mono) + a **token bridge** (legacy `--color-*`/`--sp-*`/`--dur-*`/`--radius` vars mapped onto DaisyUI theme tokens like `--color-base-100`/`--color-base-content`/`--color-base-300`, so component-scoped `<style>` blocks follow the active theme automatically). Prefer DaisyUI component classes (`btn btn-primary`, `input`, `select`, `card`, `badge badge-neutral`, `alert alert-error`, `menu`, `modal`, `dropdown`) and Tailwind utilities; don't hardcode colors — use DaisyUI tokens or the bridged `var(--color-*)`. `.card`, `.label`, `.field`, `.u-*` are thin app-specific helpers kept in app.css. Focus rings are removed app-wide (`:focus, :focus-visible { outline: none }` in app.css). Theme = `theme` cookie, applied to `<html data-theme>` at SSR via `hooks.server.ts` `transformPageChunk`; topbar toggle (light/dark) in `(app)/+layout.svelte`. Add a theme = add it to the `themes:` list (and the `THEMES` arrays in hooks + root layout load).
- Tabs for indentation; single quotes.
- Single instance, access-scoped (ADR: Auth/access): users see only workspaces/projects they can access (owner, grants, admin). Workspace ownership/grants are the sharing primitive.
- Destructive confirms: never `onsubmit` + `preventDefault()` on a `use:enhance` form (enhance's fetch fires anyway) — use the enhance callback's `cancel()`, OR the centered confirm modal. **Confirm modal**: `confirmDialog(message, { confirmLabel?, danger? })` from `src/lib/confirm.svelte.ts` returns a `Promise<boolean>`; `src/lib/components/ConfirmModal.svelte` (mounted once in the `(app)` shell, like Toaster) renders a page-centered dialog (fixed backdrop, Escape/backdrop = cancel, Enter = confirm). Pattern for an enhance form: make the trigger a `type="button"`, `await confirmDialog(...)`, then `form?.requestSubmit()` (don't await inside enhance — its fetch already fired). Task + sub-task deletes (`TaskPanel`) use this (danger, "Delete this task[ and its sub-tasks]?" / "Delete this sub-task?").
- Motion: no animation library (ADR: Cross-cutting). Use `svelte/transition`/`animate:flip` with the `--dur-fast/--dur/--dur-slow` tokens; everything ≤200ms.
- `.env` is gitignored; secrets read via `$env/dynamic/private` (server) or `process.env` (scripts).
